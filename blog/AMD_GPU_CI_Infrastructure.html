01

Scope & ownership model

This reference describes the complete CI infrastructure for a GPU stack release pipeline where firmware, kernel driver, and the ROCm compute runtime must be validated together across a fleet of GPU products. It is deliberately high-ownership: one platform engineer designs the architecture, provisions everything as code with Terraform, secures every layer (encryption, signing, audit logging), and owns reliability targets — a >95% nightly pass-rate and a <24h mean-time-to-detect on regressions.

The AMD role is explicit that nobody else on the team is currently focused on CI — so this is about setting direction and shipping the automation everything else depends on, not maintaining someone else's system.

A note on what is modeled and what is real

The architecture in §2, §4, §6 and §7 is a modeled design: directly-provisioned EC2 runners, instance profiles, CloudHSM signing, an LKG manifest. It is one coherent, defensible way to build this, and I stand behind it. But TheRock is open source, so I don't have to guess at the alternative — I can read it. Where the real system made a different call, I show both and argue the tradeoff rather than quietly retrofitting my design to match.

The single biggest divergence: my model provisions EC2 instances and registers them as runners; TheRock runs Actions Runner Controller scale sets on Kubernetes and gets an ephemeral pod per job. Neither is wrong. They optimize for different things, and §2 lays out which pressures push you toward each. A platform engineer who can only defend one of the two isn't doing architecture — they're doing preference.

Why self-hosted runners?

GitHub-hosted runners can't support firmware builds — and they can't touch AMD hardware. The build and validation environment needs:

  • NFS mounts for cross-compiler toolchains and shared build caches (ccache, DVC-managed MIOpen kernels).
  • Signing access via CloudHSM / Secrets Manager — the private key never leaves hardware.
  • Network isolation — no inbound internet, private subnets, egress via a single controlled NAT.
  • Real AMD Instinct silicon for hardware-in-the-loop, which lives in a bare-metal lab, not a cloud instance.
JD alignmentThis is exactly the "runner provisioning problem" the role calls out: standard cloud runners can't build firmware, so the work is provisioning self-hosted runners that handle NFS, signing pipelines, network permissions — and coordinating with IT on the real constraints.

In the wild: ROCm/TheRock — how runners are actually provisioned

The distinction is not decoration — one of these I'd bet a design on, the other I'd go ask a maintainer about.

TheRock does not register EC2 instances directly as GitHub Actions runners. It uses Actions Runner Controller (ARC) scale sets on Kubernetes confirmed. A listener pod watches the Actions job queue for its scale set; when a job is dispatched, ARC creates an ephemeral runner pod that executes exactly one job and is then destroyed. The evidence is in the repo's own operational documentation: runners live in an arc-runners namespace, and the documented way to inspect a running build is kubectl exec into the runner pod, not ssh into a host. For the Windows fleet, the backing cluster is AKS confirmed.

For Linux, I'd expect the equivalent on AWS — likely EKS — by parallel construction with the Windows fleet, but I want to be scrupulous here: I did not find an explicit EKS string in the repo inferred. Treat it as a hypothesis, not a finding.

One layer further down, the honest answer is I don't know, and neither does the repo. TheRock is the workload repository — it contains no runner infrastructure-as-code. Cluster and node configuration live somewhere else entirely. So whether the node layer uses Karpenter, the Cluster Autoscaler, or plain managed node groups, and whether it runs spot or on-demand, is unconfirmed inferred. What I can say architecturally is the part people miss when they skim ARC: the virtual machines don't disappear. They move. The ASG-equivalent elasticity that my model places at the runner layer, an ARC design places at the Kubernetes node layer — and the runner becomes a pod scheduled onto it.

The tradeoff, stated honestly

DimensionDirectly-provisioned EC2 modeledARC on Kubernetes in the wild
Job isolationWeak by default. Jobs share a host; a poisoned ~/.cache or a leaked /tmp file can cross job boundaries. Needs deliberate cleanup.Strong. Pod per job, destroyed on completion. Cross-job contamination is structurally prevented rather than policed.
ElasticityCoarse. Scaling means booting an instance and registering a runner — minutes.Two-tier. Pods schedule in seconds onto warm nodes; nodes scale underneath on their own clock.
Cache localityBetter by default. A persistent EBS ccache volume rides with the instance. Warm cache is the resting state.Cold by construction — an ephemeral pod inherits nothing. Which is precisely why a shared remote cache exists here (§5c). The constraint produced the better long-run design.
Operational surfaceSmall. EC2, an autoscaling group, cloud-init. Any infra engineer can debug it.Larger. You own a Kubernetes cluster, a controller, and its upgrade cadence — justified when you already run Kubernetes, expensive when you don't.
Multi-OSUniform model, separate AMIs.Windows on AKS confirmed; Linux on an AWS equivalent inferred. Two control planes is a real cost, and also the pragmatic way to get first-class Windows runners.
AMD Instinct GPU tierOrthogonal to both. MI-series test pools are their own hardware story — see §8.
My readThese optimize for different things, and the ARC choice is the right one for this repo. TheRock is a public project accepting pull requests from untrusted forks. In that setting, per-job isolation stops being a hygiene nicety and becomes the security boundary — and an ephemeral pod gives it to you structurally, rather than by remembering to clean /tmp. My EC2 model is the one I'd reach for on a private repo with a trusted contributor set, where the operational surface of a Kubernetes cluster buys you little. The right answer is a function of your threat model, not your taste. Worth noticing, too: accepting the ephemeral pod forces a shared remote cache, which is the better architecture anyway. Constraints breed good design more reliably than preferences do.

In the wild: what ROCm/TheRock runs today

This isn't a hypothetical. AMD's unified GPU-stack build system, ROCm/TheRock ("The HIP Environment and ROCm Kit"), is a public, MIT-licensed repository. It's a CMake superbuild in C++/HIP that assembles the compiler toolchain, HIP/OpenCL/core runtime, math libraries, ML libraries (MIOpen), communication libraries (RCCL), and profilers into one coherent build — the same "full recipe" this post is about.

What this stack gets right

Start here, because it's the honest headline: this is a well-run CI system. It authenticates to AWS with OIDC role assumption and no long-lived keys. It pins every third-party action to a commit SHA. It generates its build matrix as code instead of hand-maintaining a combinatorial explosion of YAML. It has already factored 38 of its 54 workflows into reusable workflow_call units. It runs a shared remote build cache. It layers gitleaks, CodeQL, Dependabot and a dedicated security pre-commit hook set. Each of those is a decision I've watched well-funded commercial teams fail to make. The sections below are about where a system this mature goes next — not about anything being wrong.

The detail that governs everything downstreamThe superbuild does not run in a docker build. It runs inside a manylinux container invoked with docker run / podman run confirmed. That single fact governs the whole caching strategy: because the compile is not a Docker layer, BuildKit's layer cache cannot touch it, and compiler caching has to be solved separately with sccache/ccache. It also means the container image is a mostly-static input that is an excellent fit for BuildKit. Two different caching problems wearing the same word. §5 and §10 pull them apart, because conflating them is how buildx gets adopted for the wrong reason and then blamed for not delivering.

Those 46 Python scripts generate JSON build configurations that GitHub Actions consumes as matrices via fromJSON() confirmed. Which GPU families build on which OS, against which Python versions, producing which artifacts, on which trigger — none of that is expressed in YAML. YAML is the output format. Python is the build system, and GitHub Actions is an executor of whatever JSON it is handed.

Let me be precise, because "12,000 lines of Python in CI" reads like an accusation and it isn't one. This layer exists because it has to. The native Actions matrix is a static cartesian product; the moment your build topology depends on the event type, the changed paths, the release channel, and a per-family capability table, the native matrix is simply not expressive enough. Every mature Actions monorepo grows this layer. TheRock's is large because its build topology is large. Config-as-code here is the right call, and the team made it before most would have.

The forward-looking opportunity is that a component this load-bearing has outgrown the tooling of a script. Four things it would get if it were treated as the product it has become:

  • A typed contract. The JSON crossing into fromJSON() is an untyped dict on the Python side and an untyped expression on the YAML side. A renamed key surfaces at 3am in the nightly rather than at review time.
  • Local reproduction. Today a contributor answers "what will CI do with my PR?" by pushing and watching. A --dry-run mode would collapse that loop to sub-second — most valuable precisely when the change is to the matrix generator.
  • Unit tests on the generator. The logic that decides what gets built is among the least-covered code in the repo. That inversion is worth correcting on its own.
  • An explicit trust boundary. fromJSON() over a string produced by an upstream job is a data/code seam. It holds today because the inputs are internal and the authors are careful — but it's the kind of seam that rewards being made structural rather than cultural.
The leverage pointNothing else in this post moves the needle as much as hardening this Python layer into a product — typed models for the build-config contract, unit tests over the generator, and a --dry-run/--event-file mode. It is unglamorous, it ships in weeks rather than quarters, it introduces no vendor and no migration, and it turns the most consequential part of the system into the most inspectable. Concrete shape in §5(a); ROI argument in §10.

Incrementality: most of the way to content-addressing

TheRock shards the superbuild into stages and reuses prior output via prebuilt_stages + a baseline_run_id, coordinated by artifact_manager.py against a declared BUILD_TOPOLOGY.toml confirmed. Notice what that is: a working, pragmatic implementation of staged incremental builds — the outcome Bazel is usually adopted to obtain — built without adopting Bazel.

The remaining gap is baseline_run_id: a human picks which prior CI run to inherit artifacts from. That's a cache key chosen by a person. It works, and it carries the failure mode you'd predict — choose a stale baseline and lower stages silently go unrebuilt.

The reflex is "so migrate to Bazel." I'd argue the opposite, and §10 makes the full case. The short version: the topology is already declared in BUILD_TOPOLOGY.toml. Deriving the stage set from that graph plus a diff needs a graph traversal, not a new build system. The goal isn't a different tool — it's to stop asking a human for the cache key.

The through-lineTheRock is Python-first, CMake/Ninja, cache-sensitive, multi-OS, multi-arch, dashboard-driven, ARC-on-Kubernetes, OIDC-federated, and already AI-in-the-loop. That is my exact toolbox — Python as primary language, multi-cloud IaC, Kubernetes runner fleets, GitHub Actions and GitLab CI, GPU provisioning, and agentic MCP tooling. I'd be productive against this stack in week one, not month three.

Hardening it: type the contract, then make it runnable offline

The fix is not a rewrite. It's giving the contract a name and a test.

build_tools/github_actions/build_config.pyTyped models for the JSON matrix contract + a --dry-run reproducerpython 
#!/usr/bin/env python3
"""Typed contract for the JSON build matrix consumed by fromJSON()."""
from __future__ import annotations

import argparse
import json
import os
import sys
from typing import Literal

from pydantic import BaseModel, Field, ValidationError

Platform = Literal["linux", "windows"]


class BuildEntry(BaseModel):
    """One row of the GitHub Actions matrix. Renaming a field here is a
    type error at review time instead of a 3am nightly failure."""

    model_config = {"extra": "forbid", "frozen": True}

    amdgpu_family: str = Field(pattern=r"^gfx[0-9a-z]+$")
    platform: Platform
    runner_label: str
    test_runs_on: str | None = None
    build_python_wheels: bool = False


class BuildMatrix(BaseModel):
    entries: tuple[BuildEntry, ...] = ()

    def to_actions_json(self) -> str:
        # Compact separators: this string is interpolated into YAML.
        return json.dumps(
            [e.model_dump(exclude_none=True) for e in self.entries],
            separators=(",", ":"),
        )


def compute_matrix(event_name: str, changed_paths: list[str]) -> BuildMatrix:
    """Pure function: (event, diff) -> matrix. No I/O, so it unit-tests."""
    if all(p.startswith("docs/") for p in changed_paths) and changed_paths:
        return BuildMatrix()  # docs-only PR: build nothing

    entries = [
        BuildEntry(
            amdgpu_family="gfx942",
            platform="linux",
            runner_label="linux-build-epyc",
            test_runs_on="instinct-mi300x",
        ),
        BuildEntry(
            amdgpu_family="gfx1100",
            platform="linux",
            runner_label="linux-build-epyc",
            test_runs_on="radeon-rx7900xtx",
        ),
    ]
    if event_name != "pull_request":
        entries.append(
            BuildEntry(
                amdgpu_family="gfx1100",
                platform="windows",
                runner_label="windows-build-arc",
                build_python_wheels=True,
            )
        )
    return BuildMatrix(entries=tuple(entries))


def main() -> int:
    ap = argparse.ArgumentParser()
    ap.add_argument("--event-name", required=True)
    ap.add_argument("--changed-path", action="append", default=[])
    ap.add_argument("--github-output", help="Path to $GITHUB_OUTPUT")
    ap.add_argument(
        "--dry-run",
        action="store_true",
        help="Print the matrix a contributor's event would produce, locally.",
    )
    args = ap.parse_args()

    try:
        matrix = compute_matrix(args.event_name, args.changed_path)
    except ValidationError as exc:
        print(f"invalid build configuration:\n{exc}", file=sys.stderr)
        return 1

    payload = matrix.to_actions_json()

    if args.dry_run or not args.github_output:
        # "What will CI do with my PR?" -- answerable without pushing.
        print(json.dumps(json.loads(payload), indent=2))
        return 0

    with open(args.github_output, "a", encoding="utf-8") as fh:
        fh.write(f"build-matrix={payload}\n")
        fh.write(f"any-builds={str(bool(matrix.entries)).lower()}\n")
    return 0


if __name__ == "__main__":
    raise SystemExit(main())

(c) Remote compiler caching: sccache against bazel-remote

Recall the two facts that make this necessary: the superbuild runs inside a container via docker run, not as Docker layers (§3), and ARC runner pods are destroyed after one job (§2). So there is no layer cache to hit and no local disk that survives. The compiler cache must be a network service.

TheRock's answer is sccache/ccache pointed at a bazel-remote server running in-cluster confirmed. Worth saying once, clearly: this is bazel-remote the cache daemon, which speaks the standard HTTP and gRPC remote-cache protocols. It implies nothing about using Bazel to build anything. The name reliably misleads — I've seen engineers spot it in a config file and conclude a Bazel migration is underway. It isn't, and §10 explains why it shouldn't be.

Keep the compiler-caching claim measuredCompiler caching for HIP device code is genuinely harder than for host C++, and I'd resist quoting a hit-rate target for hipcc that I hadn't measured on the actual tree. Multi-architecture offload bundles, embedded target IDs, and toolchain-version sensitivity all shrink the set of cacheable compilations, and the effective hit-rate on device code can sit well below what the same cache achieves on host objects. --offload-compress helps the artifact size, not the key stability. The right move is to instrument first (§10) and set the SLO from data — not to assume the host-code number generalizes.

(d) buildx / BuildKit for the container-image layer

Here is the cleanest available win, and it is available precisely because the compile happens in docker run. The build image — manylinux base, dnf packages, pip requirements, toolchain — is a separate artifact with a completely different change cadence from the C++ it hosts. It changes when someone touches requirements.txt. That is a layer-caching problem, and BuildKit is the tool for it.

These are complementary layers, not competing tools. BuildKit caches the environment; sccache caches the compilation. "buildx vs. sccache" is a category error — and an expensive one, because it leads teams to adopt buildx expecting a faster C++ compile, measure no change, and conclude the tool doesn't work.

sign_firmware.pyCloudHSM signing with provenance metadatapython 
#!/usr/bin/env python3
import subprocess, hashlib, json, os
from datetime import datetime, timezone

def sign_firmware_binary(firmware_path):
    """Sign a firmware binary via CloudHSM."""
    if not os.path.exists(firmware_path):
        raise FileNotFoundError(f"{firmware_path} not found")

    with open(firmware_path, "rb") as f:
        fw_hash = hashlib.sha256(f.read()).hexdigest()
    print(f"[*] Firmware hash: {fw_hash}")

    sign_result = subprocess.run([
        "cloudhsm_client", "sign",
        "--digest", fw_hash,
        "--key-id", os.environ["HSM_KEY_ID"],
        "--algorithm", "SHA256_RSA_PKCS",
    ], capture_output=True, text=True, check=True)

    signature = sign_result.stdout.strip()
    sig_path = f"{firmware_path}.sig"
    with open(sig_path, "w") as f:
        f.write(signature)

    metadata = {
        "firmware": os.path.basename(firmware_path),
        "hash_algo": "SHA256",
        "hash": fw_hash,
        "signed_at": datetime.now(timezone.utc).strftime("%Y-%m-%dT%H:%M:%SZ"),
        "signature_algo": "SHA256_RSA_PKCS",
        "hsm_key_id": os.environ["HSM_KEY_ID"],
    }
    with open(f"{firmware_path}.metadata.json", "w") as f:
        json.dump(metadata, f, indent=2)

    print(f"[+] Firmware signed: {sig_path}")
    return sig_path

if __name__ == "__main__":
    import sys
    sign_firmware_binary(sys.argv[1])

In the wild: OIDC federation — and why it beats my own instance-profile design

My §4 model authenticates runners with an EC2 instance profile and pulls signing material from Secrets Manager modeled. TheRock uses GitHub Actions OIDC role assumption — workflow-level role-to-assume, no long-lived AWS keys — with per-release-type IAM roles and per-release-type S3 buckets confirmed. Theirs is the better design, and it's worth saying so plainly rather than quietly redrawing my diagram to match.

An instance profile is a credential bound to a machine. Every job landing on that machine inherits it, identically. The blast radius of a compromised job is "everything the runner may ever do," and the audit trail records that rocm-ci-runner-role did it — not which workflow, on which ref, for which release channel.

Actions OIDC binds the credential to a workflow execution. GitHub mints a short-lived JWT whose sub claim encodes the repository, the ref, and optionally the environment. AWS exchanges it for temporary credentials via sts:AssumeRoleWithWebIdentity. Nothing durable is stored anywhere, and the trust policy — not a runbook, not good intentions — is what prevents the nightly job from writing to the stable release bucket.

Two OIDC mechanisms, two layers — don't conflate them

This trips people up constantly, and in an ARC-on-Kubernetes world (§2) both mechanisms are in play at once, at different layers:

MechanismIdentity is bound toWho mints the tokenUse it for
GitHub Actions OIDC

EKS IRSA / EKS Pod IdentityA Kubernetes service account on a podThe EKS cluster's own OIDC provider (IRSA), or the Pod Identity agentInfrastructure-plane access from pods — a cache server reading its S3 backing store, a controller reading SSM. The distinction matters operationally: if you grant an ARC runner pod broad S3 write access via IRSA, every job scheduled onto that pod inherits it — you've reinvented the instance profile's blast radius one abstraction layer up, while believing you'd modernized. Keep the pod's IRSA/Pod Identity role scoped to what the runner infrastructure needs (pull an image, reach the cache), and let the workflow assume its own role via Actions OIDC for anything the job needs. Right mechanism, right layer.

Note also how cleanly workload identity composes with ephemeral runners: an ARC pod has no durable machine identity worth binding a credential to. Ephemeral compute all but forces workload identity. Two independent design choices reinforcing each other — the mark of a coherent system rather than an accumulated one.

But OIDC is strictly better on audit granularity, needs zero secret rotation, and is a prerequisite once runners go ephemeral. If I were shipping my §4 design today, I'd cut the instance profile down to ssm:* for bootstrap and move every artifact-publishing action to OIDC. Supply-chain posture: what TheRock already layers

Credit where it's due. TheRock's supply-chain controls are better than most commercial CI I've audited confirmed, and they're layered rather than stacked — each control catches a class the others structurally cannot.

The gap: nothing statically analyzes the workflows themselves

Look down the "cannot catch" column. CodeQL covers the C++ and the Python. Nothing covers the ~9,480 lines of YAML — which is exactly where §5's fromJSON() string-injection boundary lives, and exactly where the most dangerous Actions misconfigurations hide: pull_request_target checking out untrusted code, ${{ github.event.* }} interpolated straight into a run: shell block, over-broad OIDC sub claims.

Two tools close this, they cost an afternoon each, and they are additive to CodeQL rather than redundant with it:

  • actionlint — correctness. Catches invalid expression syntax, unknown contexts, bad runs-on labels, and shellcheck findings inside run: blocks. It would have caught most YAML bugs I've watched teams debug via push-and-pray.
  • zizmor — security. Purpose-built Actions static analysis: template-injection sinks, dangerous pull_request_target patterns, unpinned actions, excessive permissions, credential persistence.

test_harness.pyFlash → enumerate → bandwidth → thermal → JUnitpython 
#!/usr/bin/env python3
import subprocess, re, time

class GPUTestHarness:
    def __init__(self, gpu_model, firmware_path):
        self.gpu = gpu_model
        self.firmware = firmware_path
        self.results = {"gpu": gpu_model, "tests": []}

    def flash_firmware(self):
        print(f"[*] Flashing {self.gpu}...")
        subprocess.run([
            "rocm-flash-firmware", "--device", self.gpu,
            "--firmware", self.firmware, "--verify",
        ], timeout=300, check=True)

    def test_device_enumeration(self):
        output = subprocess.check_output(["rocm-smi"]).decode()
        assert self.gpu in output, f"{self.gpu} not found in rocm-smi output"
        self.results["tests"].append({"name": "device_enumeration", "status": "PASS"})

    def test_memory_access(self):
        output = subprocess.check_output(["rocm-bandwidth-test"]).decode()
        match = re.search(r"Host to Device: ([0-9.]+)", output)
        if match:
            self.results["tests"].append({
                "name": "memory_bandwidth", "status": "PASS",
                "bandwidth_gbps": float(match.group(1)),
            })

    def monitor_thermal(self):
        print("[*] Thermal monitoring (60s)...")
        temps = []
        workload = subprocess.Popen(["rocm-benchmark", "--stress-test", "--duration", "60"])
        start = time.monotonic()
        while time.monotonic() - start < 60:
            temp_out = subprocess.check_output(["rocm-smi", "--showtemp"]).decode()
            match = re.search(r"Temperature.*: ([0-9.]+)", temp_out)
            if match:
                temps.append(float(match.group(1)))
            time.sleep(5)
        workload.wait()
        max_temp = max(temps) if temps else 0
        self.results["thermal"] = {"max_temp_c": max_temp,
                                   "status": "PASS" if max_temp < 85 else "FAIL"}

    def save_junit(self, output_path):
        tests = self.results["tests"]
        xml = '\n'
        xml += f'\n'
        for test in tests:
            xml += f'  \n'
            if test["status"] == "FAIL":
                xml += "    Test failed\n"
            xml += "  \n"
        xml += "\n"
        with open(output_path, "w") as f:
            f.write(xml)

Operational playbook & runbooks

Reliability is owned, not hoped for. These are the scenario runbooks the platform role maintains for the most common failure modes.

Runbook — pre-submit failure

Check the Actions logs first — compile error, build failure, or gate failure? If the compat check fails, verify the firmware/driver version matrix. If the security scan fails, review the findings. If logs are inconclusive, connect via SSM Session Manager (no direct SSH), check disk space and the ccache state, and reproduce locally.

Runbook — cold ccache / slow nightly

If nightly wall-clock spikes, first check ccache hit-rate and the DVC remote. A common cause is a runner replaced without its persistent cache volume re-attached, or a compiler bump invalidating the cache. Re-warm from the shared remote before assuming a real regression.

Runbook — Instinct lab fully booked

Priority queue in the workflows — nightly high, pre-submit lower. Runners check device availability and back off exponentially. Permanent fix: scale the lab with the hardware team or add a device-reservation service.

Runbook — NFS mount fails during build

On the runner, df -h /opt/rocm-tools confirms it's unmounted. Check EFS health, verify the SG allows 2049/tcp from the runner SG, and that mount targets exist in both AZs. Remount, then root-cause (AZ outage? SG change? timeout?) and update the runbook.

Modernization plan: buildx, BuildKite, Bazel — an honest evaluation

Three tools get proposed every time someone looks at a CI system this size. One of them I'd adopt this quarter. One I'd pilot before committing a dollar. One I'd actively argue against, and I'd keep arguing against it after being overruled.

The discipline that matters here is refusing to evaluate tools in the abstract. Every verdict below is a claim about this system's constraints — a CMake/HIP superbuild running inside containers via docker run, on ephemeral ARC pods, with a Python-generated matrix and a shared remote cache. Change those constraints and the verdicts change with them.

buildx / BuildKit — adopt now, for the image layer only

The technical case is in §5(d). The ROI case is this: the container image is rebuilt far more often than requirements.txt changes, and every one of those rebuilds re-downloads the same dnf and pip packages. RUN --mount=type=cache makes those downloads persist across builds. --cache-to type=registry,mode=max makes intermediate layers reusable across runners — which matters precisely because an ephemeral ARC pod has no local layer cache to inherit.

The trap to avoid is the framing "buildx will speed up our builds," followed by disappointment when the C++ superbuild takes exactly as long as before. It will. BuildKit cannot cache the compile, because the compile is not a Docker layer — it happens in docker run, after the image is built. sccache caches the compile. BuildKit caches the environment the compile runs in. Adopt buildx for what it does, and you get a clean win with no downside; adopt it expecting sccache's job, and you'll rip it out in a month having learned nothing.

RiskLow. The Dockerfile is additive (--mount directives are no-ops without BuildKit, and BuildKit is the default builder in modern Docker). Rollback is deleting two flags. Nothing about the CMake superbuild changes.

Harden the Python orchestration into a real product — adopt now

I'll restate the §3 finding because it's the thesis of this section: a 12,350-line Python layer that decides what the CI system builds is a build system. The question isn't whether to have it. The question is whether it gets the engineering rigor its blast radius deserves, or gets treated as scripts.

Three changes, in order, none of them exotic:

  1. Type the contract. Pydantic or dataclass models for the JSON build-config that crosses into fromJSON() (§5a). This is the entire interface between the two halves of the CI system, and today it is an untyped dict. A typed model turns a renamed key from a 3am nightly failure into a red CI check on the PR that renamed it.
  2. Unit-test the generator. Once compute_matrix(event, diff) is a pure function, testing "docs-only PR builds nothing" and "a push to main adds the Windows wheel job" costs a few dozen lines. The most consequential logic in the system currently has the least coverage in the repo — that inversion is the bug.
  3. Ship --dry-run / --event-file. Let a contributor pipe a saved GitHub event payload in and print the matrix that would result. This collapses the iteration loop on matrix changes from push-and-wait to sub-second, and it makes the system legible — which is worth more than the speed.
Why this beats every tool on this pageNo vendor, no migration, no new runtime, no lock-in. It ships incrementally — you can type one config model this week. And it makes every subsequent decision cheaper, including the BuildKite evaluation below, because a typed, tested, locally-runnable matrix generator is exactly what you'd need to port anywhere. Do this first regardless of what you decide about everything else.

Automate stage selection from BUILD_TOPOLOGY.toml — adopt now

Today: a human picks a baseline_run_id, and artifact_manager.py reuses prebuilt_stages from that run. That's a content-addressed incremental build with the content-addressing replaced by a person's judgment (§3).

But BUILD_TOPOLOGY.toml already declares the stage dependency graph. Given that graph plus the diff for a PR, the set of stages that must rebuild is derivable. Hash each stage's inputs — its sources, its declared upstream stages' output hashes, the toolchain version — and the baseline stops being a choice. It becomes a lookup.

This is 80% of what people want from Bazel, obtained by writing a graph traversal against a file that already exists. It is a few hundred lines of the Python from the previous section, it composes with the existing remote cache, and it requires migrating exactly nothing.

The general principleWhen a team has built most of what a famous tool provides, the reflex is to adopt the famous tool. Usually the better move is to finish what they built — because the in-house version already encodes years of hard-won knowledge about the actual constraints, and the famous tool doesn't. Bazel does not know that this HIP device code needs --offload-compress. BUILD_TOPOLOGY.toml does.

Cheap additive wins — adopt now

  • GitHub merge queue. With a superbuild this expensive, "PR was green, main is red" is a multi-hour recovery. A merge queue tests the post-merge state before merging. It costs a settings change and some CI capacity, and it eliminates an entire failure class.
  • actionlint + zizmor. Static analysis for the ~9,480 lines of YAML that CodeQL structurally cannot see (§7). An afternoon each.
  • OpenTelemetry export of step timings and sccache hit-rate. Right now "why was the nightly slow" is answered by reading logs. Emit spans per stage and a hit-rate gauge per job, and it's answered by a query. This is the prerequisite for every performance claim anyone will want to make later — including the BuildKite evaluation. You cannot pilot your way to a data-driven decision without the data pipeline first.

BuildKite — evaluate, don't pre-commit

The fit is real, and I want to state the strongest version of the case before I complicate it.

TheRock generates its build matrix in Python and hands it to Actions as a JSON string through fromJSON(). That works, but it's a workaround for a missing primitive. BuildKite's native model is buildkite-agent pipeline upload: a step runs, emits pipeline YAML/JSON on stdout, and the agent splices those steps into the running build. Dynamic pipeline generation isn't a hack there — it's the documented, intended mechanism. Everything the Python layer currently does by string-injection, it would do by design, and the fromJSON() data/code seam disappears with it.

A precision I've seen people get wrong — including me, in an earlier draftIt's tempting to say "BuildKite means giving up Kubernetes runners, because their Elastic CI Stack is EC2 + ASG." That's half true and the wrong half. The reference Elastic CI Stack is indeed EC2 + Auto Scaling Groups — but BuildKite agents also run on Kubernetes via agent-stack-k8s, which schedules a pod per job in a way that maps closely onto the ARC model. Moving to BuildKite is not a forced move to EC2/ASG. You can keep the ephemeral-pod runner architecture and change only the orchestrator. That makes the evaluation cheaper and more honest than the version of this argument I first wrote down.
What you gainWhat you pay
Dynamic pipelines as a first-class primitive; no fromJSON() string injection.A second CI system to operate throughout a migration that will not be short.
Better fan-out/fan-in ergonomics for a matrix shaped like this one.Losing GitHub-native PR integration, required checks, and merge queue.
Per-step retry and concurrency-gate semantics that Actions lacks.Per-seat licensing, against $0 for Actions on a public repository.

How I'd actually decidePilot one pipeline — the release-wheels workflow from §5(b), since it's self-contained and already parameterized. Instrument both paths with the OpenTelemetry work above. Run them in parallel for a quarter. Compare three numbers: p50/p95 pipeline wall-clock, engineer-hours spent on CI plumbing, and the count of CI-only failures. Then decide. If the numbers don't separate, the answer is no — the migration cost is certain, and the benefit was supposed to be obvious. A tool that needs a careful argument to beat the incumbent is a tool you don't need yet.

Bazel — don't migrate the superbuild

I'll be direct, because hedging here does nobody a favor. Migrating TheRock's CMake/HIP superbuild to Bazel would be a multi-quarter project with a materially negative return on investment, and I'd argue against it.

  1. HIP toolchain support is immature. Bazel's C++ rules assume a toolchain shape that HIP device compilation — multi-architecture fat binaries, gfx-target offloading, --offload-compress — does not fit without substantial custom rule authoring. You would be writing and then maintaining a HIP toolchain for Bazel. That is a project in itself, and it is not the project you were funded for.
  2. The superbuild is the accumulated knowledge. Thousands of lines of CMake encode which components build in what order, with which flags, on which platform, patched how. A migration doesn't preserve that; it re-derives it, under deadline, with the original authors unavailable. Every subtle flag that gets dropped becomes a silent correctness bug on one gfx target.
  3. The payoff is already reachable. The reason to want Bazel here is incrementality. But incrementality is obtainable from BUILD_TOPOLOGY.toml + the existing remote cache (see above), at a few hundred lines and near-zero risk. You do not need to change build systems to get a correct cache key.
  4. The cache is already there. The bazel-remote server is running today, serving sccache over a standard protocol, with no Bazel in sight. The most cited benefit of Bazel — remote caching — has been decoupled from Bazel for years. That decoupling is precisely why this system works.

Where Bazel is the right call: greenfield leaf tooling. A new Python/Go CLI, a new test-harness service, anything with no legacy build and no HIP device code. Hermetic builds and remote execution are genuinely excellent there. Use it where it wins; don't drag a decade of CMake through it to feel modern.

The pattern to take awayTwo of these six verdicts are "do the boring thing to code you already own." One is "adopt a tool that does exactly one thing well and nothing else." One is "run an experiment and let it decide." One is "no, and here's the four-point argument." That distribution is what an honest tooling evaluation looks like. A modernization plan where every candidate tool gets adopted isn't a plan — it's a shopping list, and someone else will maintain it.
