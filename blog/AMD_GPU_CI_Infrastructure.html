1. Scope & Ownership Model

This reference describes the complete CI infrastructure for a GPU stack release pipeline where firmware, kernel driver, and the compute runtime (e.g. ROCm) must be validated together across a fleet of GPU SKUs. It is deliberately high-ownership: one platform team designs the architecture, provisions everything as code with Terraform, secures every layer (encryption, signing, audit logging), and owns reliability targets (95%+ nightly pass rate, <24 hour mean-time-to-detect regression).

Three Core Responsibilities

  • Nightly integration CI — firmware + driver + runtime validated together across 10+ GPU SKUs.
  • Self-hosted runner provisioning — GitHub Actions runners that need NFS, HSM access, and network isolation, provisioned on AWS EC2.
  • AWS-aligned foundations — Terraform IaC, least-privilege IAM, encryption everywhere, compliance-ready audit trails.

What the Platform Team Owns

  • Architecture decisions: GitHub Actions vs. Jenkins, Terraform vs. CloudFormation, cloud selection.
  • Infrastructure provisioning: VPC, subnets, security groups, IAM, EC2, EFS, KMS, Secrets Manager.
  • Runner provisioning: Actions registration, NFS mounts, HSM access, auto-shutdown automation.
  • CI/CD workflows: pre-submit gates, nightly builds, release-candidate validation, LKG (last-known-good) manifest.
  • Security & compliance: firmware signing via HSM, SBOM, vulnerability scanning, audit logging.
  • Hardware integration: GPU device management, testing, benchmarking, thermal monitoring.
  • Operational excellence: observability, runbooks, failure-mode handling, cost optimization.

The CI system operates in three stages: pre-submit validation (fast feedback, 10 min SLA), nightly integration (full-stack validation), and on-demand release production (sub-hour packaging and shipping).

Why Self-Hosted Runners?

GitHub-hosted runners cannot support firmware builds. Firmware requires:

  • NFS mounts to access cross-compiler toolchains (ARM, RISC-V targets).
  • CloudHSM access for code signing (the private key never leaves hardware).
  • Network isolation (no inbound internet, private subnets only).
  • Signing material via Secrets Manager with fine-grained IAM access.

Self-hosted runners on EC2 in private subnets provide all of this while keeping the build environment under the platform team's control.

3. Terraform Infrastructure as Code

Everything is code. VPC architecture, EC2 instances, IAM roles, KMS keys, S3 buckets, CloudTrail logging — version-controlled, reviewable, reproducible, and auditable.

Module 1 — VPC (Networking)

Network design emphasizes isolation and egress control.

hcl
resource "aws_vpc" "rocm_ci" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_hostnames = true
  enable_dns_support   = true
}

# Public subnet (NAT gateway location)
resource "aws_subnet" "public" {
  vpc_id                  = aws_vpc.rocm_ci.id
  cidr_block              = "10.0.0.0/24"
  map_public_ip_on_launch = true
}

# Private subnets for runners (no public IPs)
resource "aws_subnet" "runners_private" {
  count             = 2
  vpc_id            = aws_vpc.rocm_ci.id
  cidr_block        = "10.0.${count.index + 1}.0/24"
  availability_zone = data.aws_availability_zones.available.names[count.index]
}

# NAT Gateway (centralized egress control)
resource "aws_nat_gateway" "rocm_ci" {
  allocation_id = aws_eip.nat_gateway.id
  subnet_id     = aws_subnet.public.id
  depends_on    = [aws_internet_gateway.rocm_ci]
}

# Route table for private subnets (all egress via NAT)
resource "aws_route_table" "runners_private" {
  vpc_id = aws_vpc.rocm_ci.id
  route {
    cidr_block     = "0.0.0.0/0"
    nat_gateway_id = aws_nat_gateway.rocm_ci.id
  }
}

Network Design Rationale

  • VPC CIDR 10.0.0.0/16 — 65,536 IPs for current and future growth.
  • Public subnet (10.0.0.0/24) — holds the NAT gateway only; all runners live in private subnets.
  • Private subnets (10.0.1–2.0/24) — runners across 2 AZs for high availability.
  • NAT egress — a single point of control; outbound endpoints can be whitelisted.
  • Multi-AZ — if AZ-1 fails, AZ-2 keeps building.

Right-Sizing Strategy

  • t3 for firmware — good CPU performance, reasonable cost, burstable baseline.
  • c5 for driver — CPU-optimized; the smaller c5.xlarge (not c5.2xlarge) is enough.
  • g4dn for GPU — a single instance for hardware validation; pre-submit gets lower priority.
  • gp3 volumes — ~20% cheaper than gp2 with a 3,000-IOPS baseline.
  • All encrypted — customer-managed KMS keys, not AWS-managed.
  • Auto-shutdown — idle runners terminated after 30 min via a Lambda.
  • Reserved Instances — firmware & driver builders are always-on (40–50% savings).

4. GitHub Actions CI/CD Pipelines

Two pipelines: Pre-Submit (fast feedback, blocks bad code) and Nightly (full validation, hardware tests).

Pre-Submit Validation (10 min SLA)

Triggered on pull requests that touch firmware/. Builds firmware, validates compatibility, runs security scans, and blocks incompatible code before merge.

yaml
name: Firmware Pre-Submit Validation
on:
  pull_request:
    branches: [main]
    paths: ['firmware/**']

jobs:
  build-and-validate:
    runs-on: [self-hosted, firmware-builder]
    timeout-minutes: 15
    steps:
      - uses: actions/checkout@v4

      - name: Load signing secrets from Secrets Manager
        run: |
          aws secretsmanager get-secret-value \
            --secret-id /rocm-ci/signing-keys \
            --query SecretString --output text | \
            jq -r '.fw_signing_cert' > /tmp/fw.crt

      - name: Build firmware (all targets)
        run: |
          cd firmware
          cmake -B build -DFIRMWARE_TARGETS=gfx908,gfx90a,gfx940
          cmake --build build -j$(nproc)

      - name: Generate SBOM (Software Bill of Materials)
        run: |
          syft firmware/build -o cyclonedx-json > firmware-sbom.json

      - name: Compatibility matrix check
        run: |
          python3 scripts/validate_compat.py \
            --firmware-dir ./firmware/build \
            --manifest-url s3://rocm-manifests/lkg.yaml

      - name: Static security analysis (KICS)
        run: |
          kics scan -p firmware/ -o json > kics.json
          if grep -q '"CRITICAL"' kics.json; then
            echo "::error::Critical security issues found"
            exit 1
          fi

      - name: Upload results
        uses: actions/upload-artifact@v4
        with:
          name: firmware-pre-submit-${{ github.run_id }}
          path: firmware-sbom.json
          retention-days: 7

Nightly Integration Build (Full Stack)

Runs every night at 22:00 UTC. Builds all components together, runs hardware validation across GPU SKUs, and updates the LKG manifest on success.

yaml
name: Nightly Integration Build
on:
  schedule:
    - cron: '0 22 * * *'

jobs:
  build-all-components:
    runs-on: [self-hosted, firmware-builder]
    outputs:
      firmware-digest: ${{ steps.fw.outputs.digest }}
    steps:
      - uses: actions/checkout@v4

      - name: Build firmware
        id: fw
        run: |
          cmake -B build -DFIRMWARE_TARGETS=gfx908,gfx90a,gfx940
          cmake --build build -j$(nproc)
          DIGEST=$(sha256sum build/firmware/*.bin | awk '{print $1}')
          echo "digest=$DIGEST" >> $GITHUB_OUTPUT

      - name: Push artifacts to S3
        run: |
          aws s3 cp build/firmware/ \
            s3://rocm-artifacts/nightly-$(date +%Y%m%d)/ \
            --recursive --sse aws:kms

  hardware-validation:
    needs: build-all-components
    runs-on: [self-hosted, gpu-test-runner]
    strategy:
      matrix:
        gpu: [MI300X, MI250X, MI210, MI100, MI50]
      max-parallel: 2
    steps:
      - name: Flash firmware to GPU
        run: |
          python3 scripts/gpu_firmware_update.py \
            --gpu-model ${{ matrix.gpu }} \
            --firmware ./artifacts/firmware.bin \
            --verify

      - name: Run GPU validation tests
        run: |
          python3 test_harness.py \
            --gpu-model ${{ matrix.gpu }} \
            --output junit-${{ matrix.gpu }}.xml

      - name: Upload test results
        uses: actions/upload-artifact@v4
        with:
          name: hw-test-${{ matrix.gpu }}
          path: junit-*.xml

  update-lkg-manifest:
    needs: [build-all-components, hardware-validation]
    if: success()
    runs-on: [self-hosted, firmware-builder]
    steps:
      - uses: actions/checkout@v4
        with:
          repository: org/rocm-manifest
          ref: lkg-main

      - name: Generate LKG manifest
        run: |
          python3 scripts/update_manifest.py \
            --firmware-digest ${{ needs.build-all-components.outputs.firmware-digest }}

      - name: Sign manifest with GPG
        run: |
          gpg --import /var/secrets/rocm-ci-gpg-key
          gpg --sign --detach-sign --armor manifest.yaml

      - name: Commit and push
        run: |
          git config user.name 'rocm-ci-automation'
          git add manifest.yaml manifest.yaml.asc
          git commit -m "Update LKG: ${{ needs.build-all-components.outputs.firmware-digest }}"
          git push origin lkg-main

Firmware Signing Pipeline (8-Stage Process)

  1. Build firmware on an isolated runner (no internet, /tmp as ramfs).
  2. Compute a SHA-256 hash of the binary.
  3. Sign the hash via AWS CloudHSM (VPC endpoint — the private key never leaves hardware).
  4. Attach the signature to the binary; generate an SBOM (Syft, CycloneDX format).
  5. Scan for vulnerabilities with Trivy (fail on CRITICAL).
  6. Upload to S3 (KMS-encrypted, immutable, 5-year retention).
  7. Create a detailed audit trail (build job ID, builder identity, git commits, signing timestamp).
  8. Continuous CVE scanning (daily SBOM rescan, auto-notify consumers of new CVEs).

Firmware Signing Script (Python)

python
#!/usr/bin/env python3
import subprocess, hashlib, json, os
from datetime import datetime, timezone

def sign_firmware_binary(firmware_path):
    """Sign a firmware binary via CloudHSM."""
    if not os.path.exists(firmware_path):
        raise FileNotFoundError(f"{firmware_path} not found")

    # Compute SHA-256 hash
    with open(firmware_path, "rb") as f:
        fw_hash = hashlib.sha256(f.read()).hexdigest()
    print(f"[*] Firmware hash: {fw_hash}")

    # Sign via CloudHSM (VPC endpoint, no internet)
    sign_result = subprocess.run([
        "cloudhsm_client", "sign",
        "--digest", fw_hash,
        "--key-id", os.environ["HSM_KEY_ID"],
        "--algorithm", "SHA256_RSA_PKCS",
    ], capture_output=True, text=True, check=True)

    signature = sign_result.stdout.strip()
    sig_path = f"{firmware_path}.sig"
    with open(sig_path, "w") as f:
        f.write(signature)

    metadata = {
        "firmware": os.path.basename(firmware_path),
        "hash_algo": "SHA256",
        "hash": fw_hash,
        "signed_at": datetime.now(timezone.utc).strftime("%Y-%m-%dT%H:%M:%SZ"),
        "signature_algo": "SHA256_RSA_PKCS",
        "hsm_key_id": os.environ["HSM_KEY_ID"],
    }
    with open(f"{firmware_path}.metadata.json", "w") as f:
        json.dump(metadata, f, indent=2)

    print(f"[+] Firmware signed: {sig_path}")
    return sig_path

if __name__ == "__main__":
    import sys
    sign_firmware_binary(sys.argv[1])

SBOM & Vulnerability Scanning

bash
# Generate SBOM (Software Bill of Materials)

# Scan for known CVEs
trivy fs --severity HIGH,CRITICAL \
  --format json --output trivy-results.json ./firmware/build

if grep -q '"Severity": "CRITICAL"' trivy-results.json; then
  echo "CRITICAL CVEs detected - blocking release"
  exit 1
fi

# Upload SBOM to the artifact repo (immutable storage)
aws s3 cp firmware-sbom.json \
  s3://rocm-artifacts/sbom/firmware-v1.2.0.json \
  --sse aws:kms --storage-class STANDARD_IA

6. Cloud Security: Encryption & Compliance

Encryption everywhere. All data encrypted at rest (KMS), in transit (HTTPS/TLS), with complete audit trails (CloudTrail).

Encryption at Rest (Multi-Layer)

  • S3 artifacts — KMS customer-managed key (not the default AWS key).
  • EBS volumes — KMS encryption on all runner disks.
  • EFS (NFS) — encryption at rest enabled.
  • CloudWatch Logs — KMS encryption for build output.
  • RDS (if used) — KMS encryption enabled.

KMS Key Management

hcl
resource "aws_kms_key" "rocm_artifacts" {
  description             = "KMS key for ROCm artifacts"
  deletion_window_in_days = 30
  enable_key_rotation     = true

resource "aws_kms_alias" "rocm_artifacts" {
  name          = "alias/rocm-artifacts"
  target_key_id = aws_kms_key.rocm_artifacts.key_id

Key Rotation & Management

  • Automatic annual rotation enabled.
  • Key policy restricts access to specific IAM roles.
  • CloudTrail logs all KMS operations.
  • Access can be revoked by modifying the key policy (break-glass scenario).

CloudTrail Audit Logging

A comprehensive audit trail of all API calls:

  • S3 artifact uploads, downloads, deletions (object-level logging).
  • IAM role assumptions and policy modifications.
  • KMS key access (encrypt, decrypt, policy changes).
  • Secrets Manager secret retrievals.
  • EC2 launches, terminations, security-group changes.
  • Who accessed what, when, from where, and what happened.

7. Hardware-in-the-Loop Testing

GPU validation runs on real hardware: firmware flashing, device enumeration, memory-bandwidth measurement, thermal monitoring, and performance benchmarking across the SKU matrix.

GPU Test Harness (Python)

#!/usr/bin/env python3
import subprocess, re, time

class GPUTestHarness:
    def __init__(self, gpu_model, firmware_path):
        self.gpu = gpu_model
        self.firmware = firmware_path
        self.results = {"gpu": gpu_model, "tests": []}

    def flash_firmware(self):
        """Flash firmware to the GPU with verification."""
        print(f"[*] Flashing {self.gpu}...")
        subprocess.run([
            "rocm-flash-firmware",
            "--device", self.gpu,
            "--firmware", self.firmware,
            "--verify",
        ], timeout=300, check=True)

    def test_device_enumeration(self):
        """Verify the GPU is enumerated by rocm-smi."""
        output = subprocess.check_output(["rocm-smi"]).decode()
        assert self.gpu in output, f"{self.gpu} not found in rocm-smi output"
        self.results["tests"].append({"name": "device_enumeration", "status": "PASS"})

    def test_memory_access(self):
        """Measure host-to-device bandwidth."""
        output = subprocess.check_output(["rocm-bandwidth-test"]).decode()
        match = re.search(r"Host to Device: ([0-9.]+)", output)
        if match:
            self.results["tests"].append({
                "name": "memory_bandwidth",
                "status": "PASS",
                "bandwidth_gbps": float(match.group(1)),
            })

    def monitor_thermal(self):
        """60-second sustained workload with thermal monitoring."""
        print("[*] Thermal monitoring (60s)...")
        temps = []
        workload = subprocess.Popen(["rocm-benchmark", "--stress-test", "--duration", "60"])
        start = time.monotonic()
        while time.monotonic() - start < 60:
            temp_out = subprocess.check_output(["rocm-smi", "--showtemp"]).decode()
            match = re.search(r"Temperature.*: ([0-9.]+)", temp_out)
            if match:
                temps.append(float(match.group(1)))
            time.sleep(5)
        workload.wait()
        max_temp = max(temps) if temps else 0
        self.results["thermal"] = {
            "max_temp_c": max_temp,
            "status": "PASS" if max_temp < 85 else "FAIL",
        }

    def save_junit(self, output_path):
        """Export results as JUnit XML."""
        tests = self.results["tests"]
        xml = f'<?xml version="1.0"?>\n'
        xml += f'<testsuite name="{self.gpu}-validation" tests="{len(tests)}">\n'
        for test in tests:
            xml += f'  <testcase name="{test["name"]}" classname="GPU-{self.gpu}">\n'
            if test["status"] == "FAIL":
                xml += "    <failure>Test failed</failure>\n"
            xml += "  </testcase>\n"
        xml += "</testsuite>\n"
        with open(output_path, "w") as f:
            f.write(xml)

8. Operational Playbook & Failure-Mode Runbooks

Reliability is owned, not hoped for. These are the scenario runbooks the platform team maintains for the most common failure modes — the same reasoning applies whether you are on call or explaining the design in a review.

Runbook — Pre-submit failure

Response: Check GitHub Actions logs first — compile error, build failure, or gate failure? If the compatibility check fails, verify the firmware/driver version matrix. If the security scan fails, review the KICS findings. If logs are inconclusive, connect to the runner via Systems Manager Session Manager (no direct SSH), check disk space, examine the nightly baseline, and reproduce locally. Then fix the code or investigate a toolchain version mismatch.

Runbook — GPU test cluster fully booked

Response: Implement a priority queue in the workflows — nightly high, pre-submit lower. Runners check device availability and sleep with exponential backoff when unavailable. The permanent fix is to scale the cluster with the hardware team or add a device-reservation system with blocking.

Runbook — Artifact integrity, build to consumer

Guarantee: Every artifact is signed and tracked. Firmware uses HSM signing (key never leaves hardware); kernels use a Secure Boot signature. All artifacts carry SBOMs. S3 stores them immutably with versioning. Consumers verify signatures with the public key. CloudTrail logs every access, and continuous CVE scanning flags newly disclosed issues.

Runbook — NFS mount fails during build

Response: On the runner, df -h /opt/rocm-tools confirms it is unmounted. Check EFS health with aws efs describe-file-systems. Verify the EFS security group allows 2049/tcp from the runner SG and that mount targets exist in both AZs. Remount manually, then investigate root cause (AZ outage? SG change? NFS timeout?) and update the runbook.

Runbook — Balancing cost and reliability

Approach: Right-size first (firmware builders need no GPU). Auto-shutdown idle runners after 30 min. Use Reserved Instances for baseline load (40–50% savings). Track cost per runner type and component. 95%+ availability is acceptable for pre-release validation, and savings are reinvested in monitoring and test coverage.

9. 12-Week Implementation Roadmap

Phase 1 (Weeks 1–2) — Foundation

  • VPC architecture: subnets, NAT, routing, security groups.
  • IAM roles & policies: least-privilege, scoped to resources.
  • KMS: encryption at rest for S3, EBS, EFS, CloudWatch.
  • Secrets Manager: kernel signing key, firmware signing cert.
  • CloudTrail & logging: audit trail for all API calls.

Phase 2 (Weeks 3–4) — Compute & Storage

  • EC2 runners: firmware (t3.2xl ×3), driver (c5.xl ×2), GPU test (g4dn ×1).
  • GitHub Actions registration: labels route jobs to the correct runner type.
  • EFS/NFS: /opt/rocm-tools (read-only after setup), mounted across AZs.
  • S3 artifact repository: versioning, encryption, lifecycle, logging.
  • CloudWatch monitoring: logs, metrics, alarms.

Phase 3 (Weeks 5–6) — CI/CD Pipelines

  • Pre-submit gate: 10-min SLA, build + validation + security.
  • Nightly integration: full stack, GPU SKUs, hardware tests.
  • LKG manifest: automatic update on success, GPG signing.
  • Error handling: Slack notifications, escalation procedures.

Phase 4 (Weeks 7–8) — Hardware Integration

  • GPU device management: allocate, flash, test, monitor.
  • Test harness: functional tests, benchmarks, thermal monitoring.
  • Observability: Prometheus metrics, Grafana dashboards.
  • Test the full SKU matrix: CDNA 1/2/3 families.

Phase 5 (Weeks 9–12) — Operations & Reliability

  • Cost optimization: right-sizing, auto-shutdown, Reserved Instances.
  • Runbooks: 10 common failure modes + remediation.
  • Monitoring & alerts: CloudWatch alarms, SNS notifications.
  • Disaster recovery: state backup, secrets replication, RTO 4 hours.
