Matrix Filter — 2 Jobs from 16

Every pr.yaml passes this jq expression as matrix_filter to reduce the full 16-entry matrix:

map(select(.ARCH == "amd64"))
| group_by(.CUDA_VER | split(".") | map(tonumber) | .[0])
| map(max_by([
    (.PY_VER  | split(".") | map(tonumber)),
    (.CUDA_VER | split(".") | map(tonumber))

The build pipeline produces an image. A separate workflow updates one YAML field in the config repo. ArgoCD reconciles the cluster. The two systems are decoupled — if ArgoCD is down, the image still gets built. If the build fails, ArgoCD keeps serving the last healthy image.

A5Jenkins Pipelines

Jenkinsfile.rapids-build — Key Design Points

  • agent none at pipeline level — every stage declares its own agent. CPU stages (linux-amd64-cpu16) and GPU stages (gpu-l4-latest-1 for PR/branch, gpu-a100-latest-1 for nightly) declared independently. No GPU runner idle during CPU builds.
  • CI image format — rapidsai/ci-conda:cuda12.9.1-rockylinux8-py3.12 pulled fresh per stage. Real RAPIDS CI image with conda, sccache, gha-tools pre-installed.
  • ulimit inside each Docker step — ulimit -n 1048576 set inside container steps. Global pipeline options do not propagate into Docker containers.
  • rapids-get-pr-artifact — downloads C++ artifacts for Python build and test stages. Same naming convention as GitHub Actions artifact store.

Jenkinsfile.rapids-release — 8-Stage Release Promotion

  1. Validate version format (regex YY.MM.PATCH — rejects 25.8 missing zero, 25.10 missing patch)
  2. Discover nightly packages from S3 — exits non-zero if count is 0
  3. Validate package integrity with bzip2 -t on every .tar.bz2
  4. Approval gate — Jenkins input step, restricted to rapids-release-team group, skipped if DRY_RUN=true
  5. Promote S3 nightly → stable (aws s3 sync with version string include filter)
  6. Rebuild conda channel index — without this, users cannot see new packages
  7. Tag all 7 GitHub repos via API at branch-YY.MM HEAD SHA
  8. Create GitHub Release with auto-generated release notes

spark.task.resource.gpu.amount=0.1 is the single most critical config. Default 1.0 = only 1 task per GPU at a time. On Parquet reads (I/O-bound), the GPU sits idle while waiting for disk/network and Spark cannot start the next task. Setting 0.1 allows 10 concurrent tasks — eliminates I/O blocking. Without this one change, throughput is typically 3–5× lower on ETL workloads.

Known Databricks Limitations

  • 1 GPU per executor maximum — Databricks overrides spark.executor.resource.gpu.amount to 1 regardless of setting. On p3.8xlarge (4× V100), only 1 GPU is used. Use 4 single-GPU workers instead.
  • Photon must be OFF — RAPIDS replaces Photon. They cannot coexist. Unchecking Photon is required before enabling RAPIDS.
  • CPU driver only via API — Databricks UI forces a GPU driver node. A cheaper CPU driver node requires the Databricks API or CLI.
  • Cluster logs lost on shutdown — configure log delivery to cloud storage in Advanced Options before this becomes a debugging problem.
  • Python memory pool requires cudf on workers — if spark.rapids.sql.python.gpu.enabled=true, each worker needs cudf installed via init script (not %pip install in notebook).

conda-build & Package Management

Source: Role requires conda/PyPI packaging expertise

RAPIDS packages ship as conda packages (primary) and pip wheels. The entire dependency graph is managed by a single file per repo: dependencies.yaml, processed by rapids-dependency-file-generator (DFG). RAPIDS is actively migrating from conda-build to rattler-build (Rust-based, faster). This area is core to RAPIDS build infrastructure, rooted in conda-build internals from the Anaconda ecosystem.

auditwheel — GPU Wheel Auditing and CUDA Library Exclusion

auditwheel repair scans all shared libraries linked by .so files inside a wheel, identifies any libraries that pull in symbols not in the manylinux whitelist, and bundles those libraries into the wheel's .libs/ directory with mangled sonames. This makes the wheel self-contained and portable. CUDA libraries (libcuda.so, libcudart.so, libcublas.so, and RAPIDS own shared libs like libcudf.so) must be excluded from bundling with --exclude flags for two reasons: (1) libcuda.so.1 is the user-mode driver stub — it must come from the host system's NVIDIA driver installation, not a bundled copy. Bundling a stale libcuda.so would break GPU access. (2) RAPIDS shared libs like libcudf.so are large (hundreds of MB) and are already in the wheel or in a sibling wheel. Bundling them again would double the wheel size and create runtime conflicts when two packages try to load different copies of the same CUDA library into the same process.

GPU Container Performance Diagnosis — 40% Slower Than Expected

First, check if the container is running with GPU access at all: nvidia-smi inside the container. If that works, check ulimit -n (file descriptors — should be 1048576 for sccache, but more importantly check ulimit -l for locked memory — unlimited is required for pinned memory pools). Second, check RMM pool configuration: if the container is not using PoolMemoryResource and is calling cudaMalloc per operation, that alone causes 10-100x overhead on small allocations. Third, check if cudf.pandas fallback is happening — set CUDF_PANDAS_DEBUGGING=1 and look for CPU fallback messages. Fourth, check GPU utilisation with nvidia-smi dmon — if GPU is at 20% utilisation during a workload, the bottleneck is likely PCIe bandwidth (data transfer) not compute, meaning the PoolMemoryResource is not pinning host memory for fast transfers.

Container libcuda.so.1 Error — NVIDIA CTK Configuration Issue

This is a classic container isolation issue. nvidia-smi works on the host because it uses the kernel module directly. Inside a Docker container, libcuda.so.1 (the user-mode driver stub) must be explicitly injected — it is not part of the CUDA toolkit, it comes from the NVIDIA driver installation on the host. If the container was started without --gpus all (or without --runtime=nvidia on older Docker), the NVIDIA Container Toolkit hook does not run and libcuda.so.1 is not mounted into the container. Fix: ensure the container is started with --gpus all, and that the NVIDIA Container Toolkit is installed and the Docker daemon is configured to use nvidia-container-runtime (check /etc/docker/daemon.json for "runtimes": {"nvidia": {...}}). In Kubernetes, ensure the NVIDIA device plugin is deployed and the pod spec has resources.limits.nvidia.com/gpu set, which also triggers the runtime hook.

Secure Open Source CI — copy-pr-bot and pull-request/N+ Pattern

Security. When a contributor forks rapidsai/cudf and opens a PR, GitHub Actions runs as pull_request event. In this context, the workflow has read-only token permissions and no access to secrets — intentionally, to prevent a malicious PR from exfiltrating your AWS credentials or publishing tokens. However, RAPIDS CI needs to upload build artifacts to S3 for the conda channel and access sccache in S3, which requires the AWS_ROLE_ARN. The copy-pr-bot pattern: when a contributor's PR is opened, a rapids maintainer (or the bot automatically for trusted contributors) runs a comment command that triggers copy-pr-bot to push the contributor's branch to the rapidsai repo itself as pull-request/[N]+. CI then runs as a first-party branch push with full secrets access. This is safer than pull_request_target (which runs with secrets but in the context of the base branch, causing subtle headaches). retry() {
    local n_retries=3
    local wait_secs=5
    local cmd="$@"
    local attempt=1

        echo "Attempt ${attempt}/${n_retries}: ${cmd}"
        if eval "$cmd"; then
            return 0
        fi
        attempt=$((attempt + 1))
        [ $attempt -le $n_retries ] && sleep $((wait_secs * attempt))
    done
    echo "ERROR: Command failed after ${n_retries} attempts: ${cmd}"
    return 1
}

RPATH Rewriting in conda — Why Build Prefix Paths Break Deployments

conda-build rewrites RPATH because the build environment prefix is a long path like /home/rapids/miniconda3/envs/build_1234/ that is guaranteed to not exist on any user's machine. If you ship that RPATH in a conda package, every user who installs it gets a library that can't find its dependencies at runtime — you'd see "librmm.so.26: cannot open shared object file: No such file or directory" even after a successful conda install. The correct pattern is $ORIGIN-relative RPATH: $ORIGIN tells the dynamic linker to look in the same directory as the .so file itself (or relative to it). conda-build does this as a post-processing step using patchelf. The conda prefix path is also deliberately made long with placeholder characters — the post-link script does a binary string replacement of the long placeholder with $ORIGIN, which is always shorter. If the replacement RPATH is longer than the build RPATH, the binary would overflow — which is why conda build prefixes are padded with underscores to 255 characters. This is a real gotcha when building custom conda packages outside conda-build: people set the correct RPATH during build but forget that conda post-processes it.

E1CI/CD — Technical Implementation

spark.task.resource.gpu.amount=0.1 — The Most Critical RAPIDS Spark Config

Default 1.0 means each task requires the full GPU before it can start — on a single-GPU executor, only one task runs at a time. For Parquet reads (I/O-bound), the GPU is idle while waiting for disk/network and Spark is blocked from starting the next task. Setting 0.1 allows 10 concurrent tasks per executor, matching CPU-side parallelism. For AT&T's 70 TB call records workload, this single config change produced 3–5× throughput improvement before any other tuning.

PR CI Matrix Reduction — How matrix_filter jq Reduces 16 Jobs to 2

The full shared-workflows matrix is 16 entries (2 ARCH: amd64+arm64) × (2 CUDA major: 12.x+13.x) × (4 Python: 3.10–3.13). The jq filter: (1) select(.ARCH == 'amd64') drops all arm64 (8 remain). (2) group_by CUDA major puts 12.9.1 and 13.0.2 entries into 2 groups. (3) max_by Python + CUDA patch picks highest Python at highest patch in each group. Result: 2 jobs. This is a deliberate coverage compromise — tests the most representative configuration per CUDA major. Full coverage only runs on branch and nightly builds.

RAPIDS Spark Plugin — columnarOverrides() Intercepts Operations Without Code Changes

When spark.plugins=com.nvidia.spark.SQLPlugin is set, Spark calls the plugin's columnarOverrides() method after generating the physical plan. The plugin walks the SparkPlan tree and replaces each supported CPU operator (e.g. SortExec) with a GPU equivalent (GpuSortExec). Unsupported operators stay as CPU nodes — the plugin inserts GpuColumnarToRow / RowToColumnarExec transitions automatically. To diagnose which ops run on GPU vs CPU, set spark.rapids.sql.explain=ALL — the driver log shows the full replacement plan with GpuXxx labels on GPU nodes and warnings on CPU fallbacks.

Databricks Installation — %pip install Only Installs on Driver

%pip install in a notebook only installs on the driver. When Spark submits tasks to workers, those workers have no cudf — tasks fail with ModuleNotFoundError: No module named 'cudf'. The fix is a cluster-scoped init script (/Users/<name>/rapids-init.sh) that runs as root on every node before the cluster starts. Init script installs cudf-cu12, cuml-cu12, dask-cuda from pypi.nvidia.com. For Spark RAPIDS Mode 2, also downloads the plugin JAR to /databricks/jars/. Additionally, %pip install restarts the Python kernel, losing active Spark context — another reason to avoid it.

GPU Parquet Reader Architecture and AT&T Compression Findings

Spark's native Parquet reader decodes on CPU then marshals to Spark's row format. The RAPIDS GPU Parquet reader reads compressed bytes to host memory, transfers to GPU, runs CUDA decompression kernels across all GPU cores in parallel — column batches emerge directly in cuDF format in GPU memory, no format conversion or CPU copy. AT&T's published benchmark: Parquet/Snappy outperforms Parquet/Zstandard on GPU despite Zstandard having smaller file size. The GPU Snappy decoder parallelises more efficiently than GPU Zstandard. This is the opposite of the CPU recommendation where Zstandard is preferred for network bandwidth savings.

smoke_test.sh runs as a Kubernetes Job after ArgoCD declares the rolling update healthy. If any of the 12 checks fail, the script exits 1. The deploy pipeline detects the non-zero exit from the K8s Job and runs argocd app rollback rapids-{env}. E2Databricks & AT&T — Technical Implementation

AT&T TCO — Why GPU is Cheaper Than Photon Despite Higher Per-Hour Cost

GPU VMs cost more per hour but you need far fewer of them. AT&T's GPU config used min 2 / max 16 workers vs CPU min 8 / max 12. The GPU processes groupBy aggregations on 70 TB in significantly less wall-clock time — the cluster runs fewer total hours. Cost = hours-to-complete × (nodes × cost-per-node). GPU wins because the hours reduction more than compensates for higher per-node cost. Additionally, GPU auto-scaling starts at 2 workers (not 8), so idle capacity during low-load periods costs less. The AT&T finding also holds specifically for Parquet/Snappy — Zstandard on GPU was slower than Snappy despite smaller file size, which would have changed the cost calculation.

Photon vs RAPIDS — Resolving the Coexistence Constraint

Photon must be disabled — they cannot coexist. This is a hard technical requirement, not a preference. Photon is Databricks' vectorised CPU engine; RAPIDS replaces it with GPU kernels. If ops cannot disable Photon on the existing cluster, the solution is a new cluster configuration without Photon — Databricks allows both in the same workspace. Document the performance difference and make the business case using AT&T's published results (GPU cheaper than Photon for large-scale ETL). If governance is the concern, a Databricks Cluster Policy can enforce no-Photon for designated data science clusters without affecting other teams.

CI/CD Pipeline for Automated RAPIDS Databricks Validation

Four components: (1) Init script validation — GitHub Actions triggers the Databricks API to create a cluster with the init script, runs a validation notebook importing cudf, cuml, dask_cuda and calling nvidia-smi. Fails the PR if any import fails. (2) Version matrix job — matrix across DBR 13.3, 14.3, 15.4 tests init script installs correctly on each. (3) Cluster policy as code — Terraform manages Databricks Cluster Policies pinning allowed DBR versions and enforcing no-Photon and GPU node types. dask-cudf vs RAPIDS Spark Accelerator — When and Why to Choose Each

dask-cudf is a distributed DataFrame library where each partition is a cuDF GPU DataFrame. dask-cuda creates one worker per GPU and UCX handles GPU-to-GPU data exchange via RDMA — bypassing CPU shuffle entirely. Choose dask-cudf when: (1) the workload is iterative (hyperparameter search, reinforcement learning) — Dask task graphs have lower per-iteration overhead than Spark jobs; (2) you need NVLink or InfiniBand for GPU-to-GPU shuffle — dask-cuda's UCX is direct, Spark's UCX requires additional config; (3) the pipeline mixes GPU data prep directly with PyTorch or TensorFlow training — data stays as cuDF in GPU memory without serialisation to Spark's JVM model. For AT&T's confirmed use case — simple groupBy at 3 trillion record scale — Spark RAPIDS was the right choice. For iterative ML pipelines with model checkpointing between steps, Dask is preferable.

RPATH Absolute Paths — How conda-build Prevents Runtime Failures

RPATH is a baked-in path in a .so file telling the dynamic linker where to find dependencies. Build prefix paths (e.g. /home/rapids/.../build_1234/) don't exist on user machines → "librmm.so.26: cannot open." The fix is $ORIGIN-relative RPATH: $ORIGIN = the directory of the .so file at runtime. conda-build rewrites to $ORIGIN automatically via patchelf. Build prefix is padded to 255 chars so patchelf can always fit the shorter $ORIGIN path in the binary.

sccache vs ccache — Distributed vs Local Compile Cache

ccache is local-only (disk cache on one machine). sccache is distributed — backed by S3/Azure Blob/GCS, shared across all CI runners simultaneously. RAPIDS uses sccache so every runner benefits from cache entries created by every other runner. A fresh runner starting a new matrix job finds cuDF's CMake targets already compiled by a sibling job. This is what significantly reduces compile time from cold builds (~45 min) to warm cache (~8 min, estimated). Requires ulimit -n 1048576 before sccache-dist starts (default 1024 FDs is too low).

Spark RAPIDS Qualification Tool

Source: developer.nvidia.com/blog/predicting-performance-on-apache-spark-with-gpus

The Spark RAPIDS Qualification Tool answers the hardest question before a GPU migration: which of my existing CPU Spark jobs will actually benefit from GPU acceleration? It takes Spark event logs as input and runs an XGBoost ML model (trained on NDS benchmarks and historical CPU/GPU matching runs) to predict per-job speedup without you needing to run a single GPU job first.

E1.1 How the Tool Works

INPUT: Spark event logs (Spark 2.x or 3.x CPU jobs) ↓ parse raw event log → intermediate CSV files per sqlID ↓ features extracted: • disk bytes spilled • max heap memory used • estimated scan bandwidth • per-operator query plan details • data size per stage ↓ ML MODEL: XGBoost, pre-trained on NDS benchmark (CPU + GPU historical runs) ↓ estimates speedup at individual operator level ↓ combines with heuristics for overall workload qualification OUTPUT: ① Qualified workload list (ranked by GPU migration benefit) ② Recommended Spark configs for GPU (based on cluster memory/cores + event log) ③ For CSP (AWS EMR, Databricks, Dataproc): recommended GPU cluster shape → instance type + count + GPU model
CLI usage: spark_rapids qualification --eventlogs <file-path> --platform <platform>
Install via pip: pip install spark-rapids-user-tools. Supports AWS EMR, Google Dataproc, Databricks (AWS/Azure), and on-premise.

E1.2 What Makes a Workload a Good or Bad GPU Candidate

Positive GPU indicators (accelerates well)Negative GPU indicators (may not benefit)
High-cardinality joins (hash join on large tables) Small datasets (GPU overhead > compute benefit)
Aggregations (groupBy sum, count, min, max at scale) Large amounts of data movement / shuffle between stages

Window functions on large partitions Workloads dominated by Python/Pandas UDF row-level processing

E1.3 Building a Custom Qualification Model

When pre-trained NDS models don't match your environment (different hardware, network, GPU type), build a custom model:

  1. Run identical workloads on both CPU cluster and GPU cluster
  2. Collect event logs from both runs
  3. Preprocess logs → extract features per sqlID (same CSV format as built-in tool)
  4. Train XGBoost model on (CPU_features → GPU_speedup) pairs
  5. Evaluate feature importance — disk spill, scan bandwidth, operator mix are typically top features
  6. Plug custom model into qualification tool for environment-specific predictions

Validating RAPIDS Spark Migration Value — Qualification Tool

Use the Spark RAPIDS Qualification Tool. Install spark-rapids-user-tools via pip, point it at the customer's CPU Spark event logs, and run spark_rapids qualification --eventlogs ./logs/ --platform databricks. The tool runs an XGBoost model trained on NDS benchmarks against the event log features — scan bandwidth, disk spill, operator composition, data size — and outputs a ranked list of SQL IDs by predicted GPU speedup, along with recommended Spark configs and GPU cluster shape. For jobs unlike the NDS benchmark (unusual schemas, heavy UDFs, non-standard shuffle patterns), build a custom model by running a representative sample on both CPU and GPU, collecting paired event logs, and training on the actual CPU→GPU speedup ratios. This avoids the common mistake of migrating a workload that's actually dominated by small-dataset operations or complex Python UDFs, where GPU overhead exceeds the benefit. The tool measures time spent specifically on SQL DataFrame operations — not Python driver overhead — so the predictions are realistic about where GPU acceleration actually applies.

Spark 3.0 GPU Architecture — How RAPIDS Integrates

Source: developer.nvidia.com/blog/accelerating-apache-spark-3-0-with-gpus-and-rapids

Apache Spark 3.0 introduced native GPU scheduling — a fundamental architectural shift that enabled the RAPIDS Accelerator to replace CPU operators without any user code changes. Understanding this architecture is essential for the DevOps role because it explains why each Spark config tuning parameter works the way it does.

E2.1 The Spark 2.x Problem — Why Two Clusters Were Needed

In Spark 2.x, 80% of a data scientist's time is spent on data preprocessing (ETL/feature engineering). Spark 2.x had no GPU awareness — ETL ran on CPU clusters, then data had to be transferred to separate GPU clusters for ML training. Every pipeline required two cluster configurations, data serialisation between them, and double the operational overhead.

E2.2 Spark 3.0 + RAPIDS — Single End-to-End GPU Pipeline

Spark 3.0 Single GPU Cluster — End-to-End Pipeline: Raw data (S3/ADLS/GCS) │ ▼ GPU Parquet reader (10-30x faster than CPU decoder) GPU Memory (cuDF DataFrames) │ ▼ GPU ETL: joins, aggregations, sort, filter, window ops Engineered features (still in GPU memory) │ ▼ GPU ML training: XGBoost / cuML / PyTorch / TensorFlow Model artifacts │ ▼ GPU inference (Triton Inference Server) Predictions No CPU-GPU cluster switching. No data serialisation between phases. Data stays in GPU memory from ingest through model output.

E2.3 The UCX Shuffle — Eliminating the CPU Network Bottleneck

Standard Spark shuffle routes data through CPU memory → serialise → write to local disk → send over network → deserialise → load into executor. For GPU workloads this means GPU→CPU→disk→network→CPU→GPU for every shuffle operation. RAPIDS replaces this with UCX-based GPU shuffle:

Shuffle pathHopsPerformance

RAPIDS UCX same-node (NVLink)GPU → GPU (NVLink, 600 GB/s)Near-zero latency
RAPIDS UCX cross-node (InfiniBand)GPU → RDMA NIC → GPUBypasses CPU entirely
RAPIDS UCX cross-node (TCP fallback)GPU → CPU → TCP → CPU → GPUBetter than standard (less serialization)
Verified benchmark: With 8x V100 32GB GPUs on the Criteo Terabyte click logs dataset (one of the largest public recommendation datasets), RAPIDS achieved up to 43x speedup in data preprocessing time vs an equivalent CPU Spark pipeline. CPU config: AWS r5d.24xlarge, 96 cores, 768GB RAM.

E2.4 GPU-Aware Scheduling in Spark 3.0

Spark 3.0 introduced resource scheduling for GPUs as first-class resources alongside CPU cores and memory. This is what makes spark.task.resource.gpu.amount=0.1 work — Spark now understands GPU as a schedulable resource and tracks how many GPU "slots" are available per executor.

spark.task.resource.gpu.amount=0.1        # each task uses 1/10th of the GPU resource

# Result: Spark scheduler allows 10 tasks to run simultaneously per executor
# Why this matters: Parquet reads are I/O-bound, not compute-bound
# GPU is idle 80% of the time during a Parquet read task
# With 10 concurrent tasks, GPU decode runs in parallel with I/O waits of other tasks

# The RAPIDS Accelerator's columnarOverrides() is called by Spark
# after physical plan generation to replace CPU operators:
#   SortExec           → GpuSortExec
#   ShuffleHashJoinExec → GpuShuffledHashJoinExec
#   HashAggregateExec  → GpuHashAggregateExec

RAPIDS Accelerator Performance Regression — Diagnosis and Resolution

Four most common causes in order of likelihood: (1) spark.task.resource.gpu.amount not set to 0.1 — default 1.0 means only one task runs per GPU at a time. For I/O-bound stages (Parquet reads), the GPU sits idle while a single task waits for disk/network. Setting 0.1 alone typically gives 3-5x throughput improvement. (2) High CPU fallback rate — run with spark.rapids.sql.explain=ALL and check the driver log for GpuXxx labels vs plain operator names. If you see many GpuColumnarToRow / RowToColumnarExec transitions, the cost of converting data between GPU columnar and CPU row format between stages exceeds the GPU compute benefit. Identify which operators are falling back and either configure them (spark.rapids.sql.incompatibleOps.enabled=true) or restructure the query. (3) Small dataset — GPU has fixed overhead per operation (kernel launch, CUDA context). For datasets under ~10M rows, this overhead can dominate. The Qualification Tool explicitly flags small-dataset jobs as poor GPU candidates. (4) Complex Python UDFs — Python UDFs force data off the GPU into CPU Python, execute row-by-row, then move back to GPU. One UDF per row on 1B rows will be catastrophically slow. Rewrite as a RAPIDS accelerated UDF implementing evaluateColumnar() which receives cudf.ColumnVector inputs and executes in bulk.

Multi-GPU Dask Best Practices

Source: developer.nvidia.com/blog/best-practices-for-multi-gpu-data-analysis-using-rapids-with-dask

NVIDIA's official best practices guide for multi-GPU Dask workloads covers three areas critical to production deployments: CPU/GPU backend switching, memory configuration, and UCX networking. These are real production patterns used in NVIDIA's own data curation framework (NeMo Curator).

E3.1 CPU/GPU Backend Switching — Write Once, Run Anywhere

Dask supports identical code running on CPU (numpy/pandas) or GPU (cupy/cudf) backends via a single config flag. This is the same mechanism PyTorch uses for device targeting:

# GPU backends (for RAPIDS acceleration)
dask.config.set({"array.backend": "cupy"})
dask.config.set({"dataframe.backend": "cudf"})

# CPU backends (for testing, small datasets, CI without GPU)
dask.config.set({"array.backend": "numpy"})
dask.config.set({"dataframe.backend": "pandas"})

# Or via environment variable (useful in CI/CD)
export DASK_DATAFRAME__BACKEND=cudf

# Same analysis code — no per-backend I/O calls needed
with dask.config.set({"dataframe.backend": "cudf"}):
    ddf = dd.read_parquet("s3://bucket/data/")
    result = ddf.groupby("col").agg({"val": ["sum", "min", "mean"]})
    result.compute()   # ← executes on GPU if backend=cudf, CPU if backend=pandas
Why this matters for RAPIDS DevOps: CI pipelines can run the same test suite on CPU runners (fast, cheap) using DASK_DATAFRAME__BACKEND=pandas for correctness testing, then on GPU runners with DASK_DATAFRAME__BACKEND=cudf for performance validation — without maintaining two separate test codebases. This is exactly the kind of CI design pattern the RAPIDS team needs DevOps engineers to implement.

E3.2 Optimal Dask Worker Memory Configuration

The recommended worker start command for tabular ETL workloads (filters, joins, aggregations) and deduplication (NeMo Curator):

dask cuda worker <SCHEDULER:IP>     --rmm-async \              # cudaMallocAsync — reduces memory fragmentation
    --rmm-pool-size <SIZE> \  # pre-allocate GPU memory pool (e.g., 14GB on 16GB GPU)
    --enable-cudf-spill        # spill cuDF buffers to host RAM when GPU OOM
ParameterWhat it doesWhy it matters
--rmm-async Uses cudaMallocAsync (stream-ordered allocator) instead of raw cudaMalloc Dramatically reduces memory fragmentation. Fragmentation is the #1 cause of OOM errors in long-running GPU workloads — a 14GB GPU can appear full even though only 8GB is actually in use due to fragmentation. cudaMallocAsync coalesces free blocks across CUDA streams.
--rmm-pool-size Pre-allocates a single large GPU memory pool at startup Data workloads make thousands to millions of alloc/free calls. Raw cudaMalloc/cudaFree calls are expensive (kernel launch per call). A pre-allocated pool serves sub-allocations in microseconds vs milliseconds. Typical setting: 85-90% of GPU memory (e.g., 14GB on a 16GB V100).
--enable-cudf-spill cuDF's internal host-memory spilling mechanism When a GPU operation would exceed available GPU memory, cuDF moves buffers to host RAM automatically. Preferable to OOM crash. Works best for out-of-core scenarios (dataset larger than GPU memory). Has some performance penalty but enables jobs to complete that would otherwise fail.

E3.3 UCX Networking — Essential for Dense Multi-GPU Systems

# Enable UCX protocol in LocalCUDACluster for max GPU-to-GPU performance
from dask_cuda import LocalCUDACluster
from dask.distributed import Client

cluster = LocalCUDACluster(
    protocol="ucx",               # UCX transport instead of TCP
    enable_tcp_over_ucx=True,     # TCP fallback via UCX (graceful degradation)
    enable_nvlink=True,           # NVLink for intra-node GPU-GPU (up to 600 GB/s)
    enable_infiniband=True,       # InfiniBand for inter-node (if available)
    rmm_pool_size="14GB",         # pre-allocate pool on each worker
)
client = Client(cluster)

# For multi-node: configure UCX environment
# UCX_TLS=rc,cuda_copy,cuda_ipc   ← RDMA + direct GPU memory copy + IPC
# UCX_CUDA_COPY_MAX_REG_RATIO=1.0 ← allow registering all GPU memory for RDMA
When UCX is critical vs optional: For single-node multi-GPU (NVLink-connected), UCX with enable_nvlink=True is essential — it bypasses PCIe entirely for GPU-GPU transfers. For cross-node shuffle on standard Ethernet, UCX TCP still reduces serialization overhead vs native Dask TCP. For cross-node on InfiniBand (HPC clusters), UCX RDMA is mandatory for near-linear scaling.

Multi-GPU Dask OOM — Memory Fragmentation Root Cause and Fix

This is a GPU memory fragmentation OOM — the classic symptom is nvidia-smi showing significant free memory but CUDA allocation still failing. When raw cudaMalloc (the default) frees memory, it doesn't always return contiguous blocks to the pool. After thousands of alloc/free cycles in a long ETL job, the 6GB of "free" memory exists as hundreds of non-contiguous small chunks. The next large allocation (e.g., a 2GB intermediate join result) cannot find a contiguous 2GB block and raises OOM even though 6GB is technically free. Fix: restart the dask-cuda workers with --rmm-async which switches to cudaMallocAsync — the CUDA stream-ordered memory allocator. It uses virtual memory mapping and coalesces adjacent free blocks across streams, dramatically reducing fragmentation. Additionally add --rmm-pool-size 13GB (on a 16GB GPU) to pre-allocate a single contiguous pool at startup, which makes sub-allocations behave like a traditional malloc pool. For workloads that still OOM after these fixes, add --enable-cudf-spill to allow cuDF to move buffers to host RAM automatically when GPU pressure is high.

RAPIDS 24.12 completed the migration to using CUDA pip wheels for CUDA runtime dependencies rather than expecting users to have a system CUDA installation. #!/bin/bash
# scripts/azure/cleanup-orphaned-resources.sh
# Part of the 87% Azure sprawl elimination at Trackonomy

set -euo pipefail

DAYS_THRESHOLD=${1:-30}       # resources unused for > N days
DRY_RUN=${2:-true}            # default: report only, don't delete
SUBSCRIPTION=${3:-$(az account show --query id -o tsv)}

echo "Scanning subscription: ${SUBSCRIPTION} | dry_run=${DRY_RUN}"

# 1. Unattached managed disks
echo "=== Unattached Managed Disks ==="
az disk list   --subscription "${SUBSCRIPTION}"   --query "[?diskState=='Unattached'] | [?tags.keep != 'true'].{name:name, rg:resourceGroup, size:diskSizeGb, created:timeCreated}"   --output table

# 2. Orphaned public IPs (not associated with any NIC or LB)
echo "=== Orphaned Public IPs ==="
az network public-ip list   --subscription "${SUBSCRIPTION}"   --query "[?ipConfiguration==null].{name:name, rg:resourceGroup, sku:sku.name}"   --output table

# 3. Empty resource groups (no resources, older than threshold)
echo "=== Empty Resource Groups ==="
for rg in $(az group list --query "[].name" -o tsv); do
  count=$(az resource list -g "$rg" --query "length(@)" -o tsv 2>/dev/null || echo 0)
  if [ "$count" -eq 0 ]; then
    created=$(az group show -n "$rg" --query "tags.createdDate" -o tsv 2>/dev/null)
    echo "EMPTY: $rg (created: $created)"
    if [ "${DRY_RUN}" = "false" ]; then
      az group delete -n "$rg" --yes --no-wait
    fi
  fi
done

Connecting Trackonomy Multi-Region GKE to RAPIDS GPU Cluster Architecture

The multi-region, highly secured GKE architecture in this demo directly maps to the RAPIDS GPU cluster deployment patterns (Part A). Both use GitOps (Config Sync / ArgoCD) for declarative cluster state management, Workload Identity / IRSA for credential-free pod-level cloud access, and namespace-level tenant isolation. The key difference is that RAPIDS adds GPU-aware scheduling (nvidia.com/gpu resource limits, GPU node taints) and the NVIDIA Container Toolkit runtime on top of the same security and GitOps foundation. Having built the multi-region, multi-cluster security model at Trackonomy means the RAPIDS GPU cluster security model is an extension, not a fresh start.
