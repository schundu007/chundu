The Isaac Ecosystem

4 components

NVIDIA's Isaac platform is a suite of hardware-accelerated tools for robotics development. From a DevOps perspective, each component has distinct CI implications: different base images, GPU runner requirements, build tooling, and registry configurations.

Component Purpose CI Relevance
Isaac ROS Hardware-accelerated ROS 2 packages for perception, navigation, manipulation colcon builds, NGC base images, NITROS zero-copy transport
Isaac Lab GPU-accelerated reinforcement learning simulation for robot policy training GPU runner requirements, long-running RL tests (>100 min), IsaacSim dependency
Isaac Sim Photorealistic robot simulation built on NVIDIA Omniverse Extremely large image (50 GB+), NGC auth required, no QEMU support
NITROS NVIDIA Isaac Transport for ROS — zero-copy DMA between GPU memory and ROS 2 topics CUDA version pinning, driver compatibility testing, SM arch list must match

Key Facts for CI Engineers

Isaac ROS targets: x86 with Hopper/Ada GPUs, Jetson Orin (SM 8.7), Jetson Thor (SM 10.x, Blackwell)
Base images: nvcr.io/nvidia/isaac/ros:humble-ros-base-amd64 and :humble-ros-base-arm64
NGC auth required: All pulls from nvcr.io need a valid NGC_API_KEY — username is always the literal string $oauthtoken
CUDA arch list: Must include SM 8.0, 8.6, 8.7, 9.0 for full x86 + Jetson Orin coverage; add 10.0 for Jetson Thor
NITROS requirement: CUDA driver version on runner must match the CUDA runtime in the container — driver forward compatibility is limited
Isaac Sim pull size: Pulling the IsaacSim container image can consume 40–60 GB of disk and 20+ minutes on a cold runner
NITROS zero-copy path: NITROS enables direct GPU memory → ROS 2 topic transfer without a CPU round-trip bounce. This means the CUDA context on the runner host must be active and the container must have --gpus all. A CI runner without a physical GPU cannot test NITROS code paths — it requires hardware.

Job Description Mapping

JD RequirementTechnical Translation

Common confusion: Many CI engineers treat the ROS 2 workspace as a monorepo when it is actually a multi-repo workspace managed by vcstool. Failing to understand the dependency graph means over-building (every PR rebuilds everything) or under-building (transitive dependents are not re-tested after a dependency change).

DDS and test isolation: ROS 2 uses DDS (Data Distribution Service) for pub-sub IPC. In CI, multiple test jobs on the same host can interfere via multicast DDS discovery. Set ROS_DOMAIN_ID to a unique integer per job (e.g., the GHA job matrix index mod 232) and use RMW_IMPLEMENTATION=rmw_fastrtps_cpp with unicast configuration to prevent cross-job topic pollution.

Pipeline Architecture

pull_request vs pull_request_target: Always use pull_request for CI builds, never pull_request_target. The pull_request_target event runs with write access to the base repository's secrets — a fork PR using this event type can exfiltrate your NGC_API_KEY. IsaacLab's labeler.yml uses pull_request_target with a mutable action tag — this is finding S-03 in the audit.
IsaacSim repo (isaac-sim/IsaacSim): The sister repo runs build-and-test.yml on [self-hosted, gpu] with a 240-min timeout, no Docker (native ./build.sh), and a full cache wipe on every run. Tests are commented out in the workflow — the job builds but never runs the test suite.

Multi-Architecture Build Strategy

amd64 + arm64

The biggest multi-arch pitfall for Isaac ROS is the TORCH_CUDA_ARCH_LIST build argument. Setting it to 8.0+PTX (a common default) compiles only Ampere A100 code and inserts PTX for forward compatibility — but PTX JIT compilation is slow (30+ seconds per model) and the generated code is not as fast as native CUDA. Worse, it silently excludes Jetson Orin (SM 8.7) and Jetson Thor (SM 10.x) from native compilation.

Platform Targeting Matrix

PlatformSM ArchRunner TagNotes
x86 + RTX 4090 SM 8.9 (Ada) arch:amd64, gpu:rtx4090 Developer workstation class; good for fast premerge
x86 + A100 SM 8.0 (Ampere) arch:amd64, gpu:a100 Production training/inference class; required for Isaac Lab tests
Jetson Orin SM 8.7 (Ampere) arch:arm64, board:orin 64 GB unified memory variant recommended for CI; native arm64 builds only
Jetson Thor SM 10.x (Blackwell) arch:arm64, board:thor Next-gen platform; begin CI runner provisioning now

Correct CUDA Arch List

Dockerfile — ARG at top of file
ARG TORCH_CUDA_ARCH_LIST="8.0;8.6;8.7;8.9;9.0;10.0+PTX"
# 8.0  = A100 (Ampere)
# 8.6  = RTX 3090 (Ampere)
# 8.7  = Jetson Orin (Ampere mobile)
# 8.9  = RTX 4090 (Ada)
# 9.0  = H100 (Hopper)
# 10.0 = Jetson Thor / RTX 5090 (Blackwell)
# +PTX = forward-compatible PTX for future GPUs

Multi-Stage Dockerfile Pattern

Dockerfile — multi-stage builder/runtime split
# ── BUILD STAGE ──────────────────────────────────────────────────────────────
FROM nvcr.io/nvidia/isaac/ros:humble-ros-base AS builder
ARG CUDA_VERSION=12.4
ARG TORCH_CUDA_ARCH_LIST="8.0;8.6;8.7;8.9;9.0;10.0+PTX"

WORKDIR /workspace/src

# CRITICAL: COPY requirements before source — preserves pip layer on src changes
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .                             # source after deps — deps layer is cached
RUN colcon build --symlink-install \
    --cmake-args \
      -DCMAKE_BUILD_TYPE=Release \
      -DTORCH_CUDA_ARCH_LIST="${TORCH_CUDA_ARCH_LIST}"

# ── RUNTIME STAGE ─────────────────────────────────────────────────────────────
FROM nvcr.io/nvidia/isaac/ros:humble-ros-base AS runtime
COPY --from=builder /workspace/install /workspace/install
# Build tools (gcc, cmake, python-dev) are NOT present in this image
# Result: 1.5-3 GB smaller than single-stage, smaller attack surface

LABEL org.opencontainers.image.source="https://github.com/myorg/isaac-ros"
LABEL org.opencontainers.image.revision="${GIT_SHA}"
LABEL org.opencontainers.image.created="${BUILD_DATE}"

ENTRYPOINT ["/ros_entrypoint.sh"]
CMD ["bash"]

QEMU is not acceptable for arm64 production builds: QEMU software emulation is 5–10x slower than native ARM execution. A postmerge multi-arch build that takes 30 minutes on native Jetson Orin runners takes 90+ minutes with QEMU. Provision native Jetson Orin or AWS Graviton runners tagged arch:arm64 and use them for the arm64 build leg.

Multi-Repo Workspace Management

vcstool + colcon

Isaac ROS is composed of dozens of independent repositories. A CI workspace must import them all in a reproducible, auditable way. The vcstool utility manages this via a .repos YAML file. The critical rule: every entry must be pinned to a full commit SHA, never a branch name. Branch names are mutable — they can be force-pushed and will silently change your build.

vcstool .repos File Pattern

repos/isaac_ros.repos — commit SHA pinning
repositories:
  # RULE: always pin to full commit SHA, never to a branch or tag
  # Tags can be force-pushed; SHAs are immutable
  isaac_ros_common:
    type: git
    url: https://github.com/NVIDIA-ISAAC-ROS/isaac_ros_common
    version: abc1234def5678901234567890abcdef12345678

  isaac_ros_nitros:
    type: git
    url: https://github.com/NVIDIA-ISAAC-ROS/isaac_ros_nitros
    version: 9f2a3b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0a

  isaac_ros_image_pipeline:
    type: git
    url: https://github.com/NVIDIA-ISAAC-ROS/isaac_ros_image_pipeline
    version: 1a2b3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b

Affected-Package Detection

Running colcon build on the full workspace for every PR wastes GPU runner time. The correct approach detects which packages changed (and their transitive dependents) and builds only those.

scripts/affected_build.sh
#!/bin/bash
# Detect which colcon packages were affected by this PR

# 1. Get changed files vs merge base
CHANGED_FILES=$(git diff --name-only origin/main...HEAD)

# 2. Map changed files to package names (find nearest package.xml)
CHANGED_PKGS=$(python3 scripts/find_affected_packages.py "$CHANGED_FILES")

echo "Affected packages: $CHANGED_PKGS"

# 3. Build affected packages AND everything that depends on them
colcon build \
  --symlink-install \
  --cmake-args -DCMAKE_BUILD_TYPE=Release \
  --packages-up-to $CHANGED_PKGS

# 4. Test affected packages and their dependents
colcon test --packages-above-and-dependencies $CHANGED_PKGS

Cache Layer Strategy

LayerCache KeyBackendHit Rate Target
pip dependencies Hash of requirements.txt + platform arch S3 bucket via pip cache >95%
colcon build artifacts Git commit SHA + platform + CUDA version S3 + sccache for CUDA objects >80%
Docker layers Dockerfile hash per stage (builder vs runtime) Registry cache (type=registry) on ECR or NGC >90%
rosdep cache Hash of rosdep.yaml + distro GHA cache action >90%
apt package lists Date (weekly) + OS version GHA cache action ~70% (weekly expiry)
GHA cache 10 GB limit: GitHub Actions' built-in actions/cache has a 10 GB total limit shared across all branches and all workflows in a repository. Docker layer caches will bust this limit frequently on Isaac ROS workspaces. Switch Docker layer caching to type=registry cache on an ECR or NGC private registry — no size limit, much higher hit rates.

Release Engineering

target: SLSA L3
Prescriptive architecture: The pipeline below describes the recommended target state — not what currently exists in isaac-sim/IsaacLab. The real repo has no image-signing, no SBOM, no SLSA provenance, and no formal staging-to-production promotion. It operates at SLSA Level 0. The diagram is the design you should build.

A production release pipeline for Isaac ROS images must provide signed images, attached SBOMs, and verifiable provenance — the requirements for SLSA Level 3. Images move through three stages: dev (every postmerge commit), staging (release candidates), and production (the NGC public registry).

Stage 1 — Dev
Postmerge Image Push

Every merge to main triggers a multi-arch build and pushes to the dev registry with a 7-day retention policy. These images are signed and scanned but not promoted.

nvcr.io/myorg/isaac-ros-dev 7-day retention cosign signed
Stage 2 — Staging
Release Candidate Promotion

A release candidate is promoted from dev to staging by a manual workflow dispatch or a rc/* tag push. Staging images undergo hardware-in-loop validation on a physical Jetson Orin before being cleared for production.

nvcr.io/myorg/isaac-ros-staging 30-day retention HIL validated
Stage 3 — Production (NGC)
NGC Publish

After staging validation, images are promoted to the NGC production registry. SBOM is attached, provenance attestation is recorded, and the release is permanent (no TTL).

nvcr.io/nvidia/isaac/ros permanent SBOM attached SLSA L3

Signing, SBOM, and Provenance Commands

release pipeline — post-build steps
# 1. Sign the image digest with cosign
cosign sign --key cosign.key \
  nvcr.io/myorg/isaac-ros@${IMAGE_DIGEST}

# 2. Generate SBOM in SPDX-JSON format (syft)
syft nvcr.io/myorg/isaac-ros:${TAG} \
  -o spdx-json > sbom.json

# 3. Attach SBOM to the image
cosign attach sbom \
  --sbom sbom.json \
  nvcr.io/myorg/isaac-ros:${TAG}

# 4. Generate and attach build provenance (SLSA Level 3)
cosign attest \
  --predicate provenance.json \
  --type slsaprovenance \
  --key cosign.key \
  nvcr.io/myorg/isaac-ros@${IMAGE_DIGEST}

# 5. Verify before promoting to NGC
cosign verify \
  --certificate-identity-regexp '.*github.com/myorg/isaac-ros.*' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  nvcr.io/myorg/isaac-ros:${TAG}
SLSA Level 3 requirements: (1) Hermetic builds — build process reads only declared inputs; (2) Isolated runners — no shared state between builds; (3) Provenance attestation — cryptographically signed build record including inputs, outputs, and build environment; (4) Signed outputs — all published artifacts signed with a verifiable key or OIDC keyless signing.

Build Time & Caching

8 findings
B-01
COPY source before pip install — busts deps cache on every PR
Source COPY . . appears before pip install -r requirements.txt. Since any source file change invalidates Docker layer cache at that point, the entire pip install layer is re-executed on every PR — even when only a comment changed.

docker/Dockerfile.curobo:87, 131-132

Fix: COPY requirements.txt . → RUN pip install → COPY . .
Impact if unresolved: 8–12 min wasted per PR run on pip dependency reinstall.
CRIT
B-02
GHA cache backend hits 10 GB cap — needs registry cache
The Docker build uses actions/cache as the layer cache backend. This has a 10 GB total limit shared across all branches and workflows. Isaac ROS base images alone exceed 10 GB, causing constant cache eviction and near-zero effective hit rates.

.github/actions/docker-build/action.yml

Fix: Switch to --cache-from type=registry,ref=nvcr.io/myorg/cache:buildcache and --cache-to type=registry,...,mode=max. Use a private ECR or NGC registry for the cache target.
Impact if unresolved: Effective Docker cache hit rate <10%, 35+ min per build that could be <5 min on warm cache.
CRIT
B-03
No path filters — Markdown edit triggers full GPU container rebuild
Every workflow triggers on every push and pull_request without paths-ignore or paths filters. A typo fix in README.md triggers a 180-minute GPU container build (timeout-minutes: 180 in the actual build.yml) and consumes a self-hosted GPU runner slot.

.github/workflows/build.yml (all workflows)

Fix: Add to every workflow that builds code:
paths-ignore: ['**.md', 'docs/**', '*.txt', 'LICENSE', 'CHANGELOG*']
Impact if unresolved: Up to 180 min of GPU time wasted per non-code PR; queue starvation for genuine build PRs.
CRIT
B-04
License-check installs full IsaacSim stack to enumerate licenses
The license-check.yaml workflow uses pip-licenses, which installs the complete production dependency tree (including IsaacSim at 40+ GB) to introspect license metadata. This runs in the PR critical path and takes 20+ minutes.

.github/workflows/license-check.yaml

Fix: Resolve licenses from requirements.txt directly using pip-audit or pip-licenses --from=mixed in a lightweight Python venv without installing the actual packages. Cache the license list output keyed by a hash of requirements.txt.
Impact if unresolved: 20–25 min overhead in PR critical path for a check that could complete in <60 seconds.
CRIT
B-08
QEMU emulation for arm64 instead of native runner
Postmerge multi-arch builds use QEMU software emulation for the arm64 build leg. QEMU emulation is 5–10x slower than native execution and cannot run GPU code paths. The postmerge arm64 build takes ~90 minutes vs ~30 minutes on native Jetson Orin runners.

.github/workflows/postmerge-ci.yml

Fix: Provision native Jetson Orin CI runners (or AWS Graviton as proxy) tagged [self-hosted, arch:arm64]. Use a build matrix with explicit platform selectors for each leg.
Impact if unresolved: ~60 min extra per postmerge publish; QEMU cannot test NITROS or other GPU code paths.
CRIT
B-05
lfs:true on every checkout — repo has zero LFS files
Every actions/checkout step sets lfs: true, which invokes Git LFS batch operations on every checkout. The repository contains no LFS-tracked files, making this a pure overhead with no benefit.

.github/workflows/*.yml (all checkout steps)

Fix: Remove lfs: true from all checkout steps. Add it back only if LFS tracking is introduced.
Impact if unresolved: 3–5 s wasted per job, compounded across all parallel jobs and all PRs.
MED
B-06
fetch-depth:0 on every checkout — pulls full git history
All checkout steps use fetch-depth: 0 (full history). For build and test jobs, only the working tree is needed. Full history adds 15–40 s per checkout on repositories with long git history.

.github/workflows/*.yml (all checkout steps)

Fix: Use fetch-depth: 1 for all build, test, and lint jobs. Use fetch-depth: 0 only for semantic-release, changelog generation, or jobs that need git log.
MED
B-09
cuRobo CUDA compile not using sccache
CUDA compilation (via nvcc) is the most expensive build step in the Isaac ROS stack. The pipeline does not configure sccache as a compiler wrapper, meaning every CUDA source file is recompiled from scratch on cache miss rather than retrieved from a shared object cache.

docker/Dockerfile.curobo, .github/actions/docker-build/action.yml

Fix: Install sccache, set RUSTC_WRAPPER=sccache, configure SCCACHE_BUCKET=s3://your-bucket, and pass -DCMAKE_CUDA_COMPILER_LAUNCHER=sccache to cmake. Expect 60–80% hit rate on warm cache.
MED

Image Size

4 findings
I-01
No multi-stage builder/runtime split — build tools in final image
The final production image contains build-time tools: gcc, g++, cmake, python3-dev, build-essential, and all pip build wheels. These are never needed at runtime. Keeping them inflates the image by 1.5–3 GB and adds unnecessary attack surface.

docker/Dockerfile.base, docker/Dockerfile.curobo

Fix: Use a FROM ... AS builder stage for compilation, then FROM ... AS runtime with only COPY --from=builder /workspace/install .. The runtime stage should not install any compiler or Python development headers.
Impact if unresolved: 1.5–3 GB larger images; slower pulls on cold runners; broader CVE surface area in production deployments.
CRIT
I-03
TORCH_CUDA_ARCH_LIST=8.0+PTX — excludes Jetson Orin and Thor
The default TORCH_CUDA_ARCH_LIST=8.0+PTX compiles native code only for SM 8.0 (A100) and inserts PTX for all other GPUs. Jetson Orin (SM 8.7) and Jetson Thor (SM 10.x) fall back to PTX JIT at first run, adding 30–60 seconds of JIT compilation per model load and producing slower code than native SASS.

docker/Dockerfile.curobo ARG line, .github/actions/docker-build/action.yml

Fix: ARG TORCH_CUDA_ARCH_LIST="8.0;8.6;8.7;8.9;9.0;10.0+PTX"
The +PTX at the end still provides forward compatibility for future GPUs.
CRIT
I-02
Dockerfile.base and Dockerfile.curobo are 90% duplicated
Both Dockerfiles begin with the same base image, the same apt-get install block, the same ROS workspace setup, and the same Python environment configuration. Maintenance changes must be applied in two places; drift between the files causes subtle environment differences in CI.

Security

7 findings
S-01 / S-02
docker login -p $NGC_API_KEY leaks secret to process list
Both the composite action and the postmerge workflow use docker login -p $NGC_API_KEY. The -p flag puts the secret directly in the process argument vector, which is visible to any process that can read /proc/<pid>/cmdline on the same host. This is a well-known secret leak pattern.

.github/actions/docker-build/action.yml:37, .github/workflows/postmerge-ci.yml:64

Fix (everywhere): 
echo "$NGC_API_KEY" | docker login nvcr.io \
  --username '$oauthtoken' \
  --password-stdin
The --password-stdin flag reads the password from stdin, keeping it out of the process argument list. Note that the username for NGC is the literal string $oauthtoken (including the dollar sign), not a variable.
CRIT
S-03
pull_request_target with mutable action tags — secret exfiltration risk
The labeler.yml workflow uses the pull_request_target event, which runs with write permissions from the base repository (including access to secrets). Combined with third-party actions pinned to mutable version tags (@v6), a compromised upstream action maintainer can force-push a tag and inject code that exfiltrates NGC_API_KEY during the next PR from any fork.

.github/workflows/labeler.yml

Fix: Switch the labeler workflow to use pull_request instead of pull_request_target. If write access to the PR is required (e.g., to apply labels), use a separate workflow_run triggered workflow with minimal permissions. SHA-pin all third-party actions.
CRIT
S-04
Self-hosted GPU runner executes fork PR code without a trust gate
The main build.yml workflow runs on [self-hosted, gpu, x86] runners for all pull requests, including those from forks. A malicious contributor can submit a PR that reads environment variables or files from the runner host, exfiltrating secrets, runner SSH keys, or NGC API credentials.

.github/workflows/build.yml

Fix: Gate all self-hosted runner jobs with an explicit trust check: 
if: github.event.pull_request.head.repo.full_name == github.repository
For fork PRs, run only on GitHub-hosted runners with no secrets. Require a maintainer approval comment to trigger GPU jobs on fork PRs.
CRIT
S-07
All third-party actions pinned to mutable version tags
Every third-party action reference uses a mutable semver tag: actions/checkout@v4, actions/upload-artifact@v2, actions/labeler@v6. A git tag can be force-pushed to point to an entirely different commit without changing the tag name, silently changing the code your pipeline executes.

.github/workflows/*.yml (all workflow files)

Fix: Pin every third-party action to its full commit SHA: 
uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683  # v4.2.2
Use Renovate or Dependabot to keep SHAs up-to-date automatically.
CRIT
S-08
No CodeQL/SAST, Trivy/SBOM, cosign, Dependabot, or Renovate
The repository has zero supply-chain controls. There is no SAST scanning, no container image scanning, no signed images, no SBOM, no automated dependency update bot, and no provenance attestation. This is a SLSA Level 1 posture for a production robotics platform.

Repository-wide — no .github/workflows/codeql.yml, trivy.yml, renovate.json

Fix (priority order):
1. Add renovate.json or Dependabot config to auto-update action SHAs and pip dependencies.
2. Add trivy.yml workflow to scan images for CVEs on every postmerge push.
3. Add codeql.yml for C++ and Python static analysis.
4. Add cosign image signing and SBOM attachment in the postmerge pipeline.
5. Add SLSA provenance attestation to reach SLSA Level 3.
CRIT
S-05
Container runs as root with --network=host
Test containers run as UID 0 (root) and with --network=host. A container process exploiting a vulnerability can access the host network directly and write to host paths mounted as volumes.

.github/workflows/build.yml (container options)

Fix: Add --user 1000:1000 to container options. Use --network=bridge for test containers; reserve --network=host only for jobs that genuinely require host network access (e.g., NITROS transport tests).
MED
S-06
DOCKER_TLS_CERTDIR="" disables TLS on Docker socket
Setting DOCKER_TLS_CERTDIR="" in the DinD (Docker-in-Docker) service disables TLS on the Docker socket. Any process on the runner host that can reach the Docker socket can execute arbitrary containers without authentication.

.github/workflows/build.yml (services: docker section)

Fix: Set DOCKER_TLS_CERTDIR to a valid certificate directory path (e.g., /certs) and configure the client to use TLS. Alternatively, use the Docker socket from the host runner rather than DinD.

Colcon & Ament Build System

Build reference

Colcon is the meta-build system that orchestrates ROS 2 package builds. It discovers packages by finding package.xml files, resolves the dependency graph, and invokes the appropriate build tool (ament_cmake, ament_python, cmake, or plain Python setuptools) for each package in topological order.

Essential Colcon Commands

CommandWhat It DoesWhen to Use in CI
colcon build Build all discovered packages in topological dependency order Full workspace build in postmerge and release jobs
colcon build --packages-select pkg_a pkg_b Build only the specified packages (no dependencies) Targeted rebuild of known-affected packages only
colcon build --packages-up-to pkg_a Build pkg_a and all packages it depends on (transitive closure of dependencies) Preferred for premerge — build the changed package and everything it needs
colcon build --packages-above pkg_a Build all packages that depend on pkg_a (reverse dependency closure) After changing a shared library — rebuild all consumers
colcon build --packages-above-and-dependencies pkg_a Build pkg_a's full upward closure plus all their dependencies Comprehensive affected-package rebuild in premerge
colcon test Run all test suites (ament, pytest, gtest) for all packages After build step; outputs JUnit XML to build/*/test_results/
colcon test-result --all --verbose Aggregate and display test results from all packages After colcon test; fail CI if any test failed
colcon graph Print the package dependency graph (tree or dot format) Debugging dependency issues; input to affected-package scripts
colcon list List all packages discovered in the workspace with their paths Validate workspace completeness after vcs import

package.xml Dependency Types

Rosdep resolves different dependency types differently. The CI engineer must understand these to correctly configure the install step and avoid over-installing (pulling in build tools into the runtime image).

package.xml — format 3 with all dependency types
<package format="3">
  <name>my_isaac_ros_pkg</name>
  <version>1.0.0</version>

  <!-- Both build AND runtime dependency -->
  <depend>rclcpp</depend>
  <depend>sensor_msgs</depend>

  <!-- Build-time only — NOT installed in runtime image -->
  <build_depend>ament_cmake</build_depend>
  <build_depend>rosidl_default_generators</build_depend>

  <!-- Runtime only — not needed to compile -->
  <exec_depend>python3-numpy</exec_depend>
  <exec_depend>isaac_ros_nitros</exec_depend>

  <!-- Test only — excluded from both runtime and build images in CI -->
  <test_depend>ament_lint_auto</test_depend>
  <test_depend>ament_cmake_gtest</test_depend>

  <export>
    <build_type>ament_cmake</build_type>
  </export>
</package>
--symlink-install flag: Using colcon build --symlink-install creates symlinks from the install/ directory back to the source files instead of copying them. This means Python scripts and launch files can be edited in place without rebuilding. Use this in development CI; do NOT use it in release/production builds where the install directory must be self-contained and portable.

COLCON_EXTENSION_BLOCKLIST

In CI, you often want to exclude packages that should not be built (e.g., example packages, tutorial code, or packages with external hardware dependencies). The COLCON_EXTENSION_BLOCKLIST environment variable provides a coarser mechanism, but for per-package exclusion, use --packages-ignore:

CI script — excluding non-essential packages
colcon build \
  --symlink-install \
  --cmake-args -DCMAKE_BUILD_TYPE=Release \
  --packages-ignore \
    isaac_ros_examples \
    isaac_ros_tutorials \
  --packages-up-to isaac_ros_nitros isaac_ros_image_proc

Quick Diagnostic Checklist — Pipeline Failure Triage
  1. Runner OOM? Check ci_gpu_utilization metric and GHA job log for Killed or CUDA_ERROR_OUT_OF_MEMORY. Fix: add swap, increase runner memory, or reduce --parallel-workers.
  2. Cache miss? Check ci_cache_hit_rate. If <50%, check that cache keys include all relevant inputs (requirements hash, platform, CUDA version).
  3. rosdep failure? Usually a new apt package dependency added to package.xml that is not yet in the rosdep database. Fix: add an override in rosdep.yaml or file an upstream PR.
  4. NITROS transport failure? Check that the runner has --gpus all and that the CUDA driver on the host matches the CUDA runtime in the container (check nvidia-smi vs nvcc --version).
  5. Flaky test? Check ci_test_flake_rate. Re-run the test 5 times in isolation. If it passes 4/5, add @pytest.mark.flaky(reruns=3) as a temporary measure and file a stability bug.
  6. NGC pull failure? Verify NGC_API_KEY secret is set and not expired. Check that authentication uses --password-stdin and username is the literal $oauthtoken.
  7. QEMU timeout? This is finding B-08. QEMU is too slow for production arm64 builds. Provision native Jetson Orin or Graviton runners.
