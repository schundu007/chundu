Hard System Design
Design the infrastructure for an e-commerce shopping cart that handles 4 million concurrent users during prime time on GCP.

Approach Framework

At 4M concurrent users, we're looking at approximately 200K-400K requests/second at peak. The shopping cart is a write-heavy, session-critical service - any data loss means revenue loss. The design must prioritize low-latency reads/writes, horizontal scalability, and strong consistency for the cart state.

Key Design Decisions

1. Cart State in Memorystore Redis Cluster: Shopping carts are ephemeral, high-throughput, and latency-sensitive. A 30-shard Redis Cluster delivers sub-millisecond reads/writes with 100K+ ops/sec per shard.

2. Cloud Spanner for Orders and Inventory: Spanner provides externally consistent, globally distributed SQL with automatic sharding. For inventory, Spanner's TrueTime-based transactions prevent overselling.

3. GKE Autopilot with Aggressive HPA: HPA configured on custom metrics (requests-per-second from Cloud Monitoring) rather than CPU alone.

Capacity Math: 4M concurrent users x ~1 request every 10 sec = 400K RPS peak. Cart Service: 1000 pods x 400 RPS each = 400K RPS capacity. Redis: 30 shards x 15K writes/sec = 450K write ops/sec.

Pro Tip: Always start with back-of-envelope math. Interviewers want to see you quantify the problem before drawing boxes.

Phase Details

Phase 1 - Assessment: Catalog CUDA versions, frameworks, multi-node communication patterns. Map A100 40GB vs 80GB usage to GCP machine types (a2-highgpu vs a2-ultragpu).

Phase 2 - Foundation: Shared VPC with Dedicated Interconnect. VPC Service Controls around BigQuery and GCS. Terraform modules for reproducible GPU cluster provisioning.

Phase 3 - Pilot: GKE cluster with A2 GPU node pool. NVIDIA GPU Operator for driver management. Deep Learning Container Images.

Phase 4 - Scale: Kueue for multi-tenant scheduling. Dynamic Workload Scheduler for cost optimization. Consider A3 Mega (H100) or TPU v6e for better performance.

Data Transfer: Less than 10TB: gsutil parallel uploads. 10-100TB: Storage Transfer Service. Greater than 100TB: Transfer Appliance.

Design Decisions

1. Separate Node Pools: Training (GPU/TPU), inference (L4), and platform services (CPU) each get dedicated pools with taints and tolerations.

2. Kueue for Scheduling: ClusterQueue with ResourceFlavors for GPU types. Priority classes: production > research > dev.

3. Autoscaling: Cluster Autoscaler + HPA + Node Auto-Provisioning. Scale GPU pool to zero when idle.

4. Checkpointing: GCS FUSE CSI Driver for transparent GCS access. Cloud Managed Lustre for high-throughput data loading.

Hard TPU/GPU Design
A customer needs to run training on a 70B parameter LLM. How would you architect the TPU/GPU infrastructure on GCP?

Sizing Analysis

A 70B parameter model in BF16 requires approximately 140GB memory for weights. With optimizer states, gradients, and activations: 560-700GB total. This requires model parallelism across multiple devices.

Options

OptionConfigurationUse Case
GPU Path (A3 Mega)4x a3-megagpu-8g = 32x H100 = 2,560GB HBM3PyTorch teams, CUDA dependencies
TPU Path (v6e)256-chip pod = 8,192GB HBM3JAX teams, best perf/$

Parallelism Strategy: Tensor Parallelism (TP=8) within node + Pipeline Parallelism (PP=4) across nodes for GPUs. FSDP across all chips for TPUs.

Cost (monthly, 24/7): 4x A3 Mega on-demand: ~$254K. With 3yr CUD: ~$160K. 256-chip TPU v6e on-demand: ~$140K. TPU with CUD: ~$80K.

Hard Multi-Cloud
How would you set up a secure multi-cloud architecture where training happens on GCP TPUs but inference runs on another cloud?

Key Design Points

1. Model Export: Train in JAX on TPUs, export to ONNX or SavedModel for framework-agnostic inference.

2. Connectivity: HA VPN for control plane. Storage Transfer Service for large model transfers (GCS to S3).

3. Identity (Zero Trust): Workload Identity Federation with OIDC. No long-lived credentials cross cloud boundaries.

4. Model Sync Pipeline: Training completes -> validated -> Vertex AI Model Registry -> export to ONNX -> copy to S3 -> canary deployment -> full rollout.

Security: Encrypt in transit (TLS 1.3). CMEK in both clouds. VPC Service Controls on GCP. Network policies in both clusters.

Medium Hybrid Cloud
Your company has on-premises infrastructure and wants a hybrid cloud solution. Design the integration using GCP, ensuring seamless data synchronization.

Key Components

Network: Dedicated Interconnect (2x 10Gbps, redundant) with Cloud Router BGP. Cloud DNS peering zones for bidirectional resolution.

Data Sync: Datastream for CDC from Oracle/MySQL to BigQuery. Storage Transfer Service for NFS to GCS. Pub/Sub for event-driven sync.

Identity: Workforce Identity Federation (AD/SAML to GCP IAM). Workload Identity Federation for on-prem K8s.

Security: VPC Service Controls perimeter. Cloud KMS with External Key Manager (keys stay on-prem).

Hard Framework Migration
A customer is using PyTorch with CUDA dependencies. Can they use TPUs? What migration path would you recommend?

Migration Paths

Path A - PyTorch/XLA (Low CUDA dependency): Install torch_xla, replace .cuda() with xm.xla_device(). Most standard ops work. Best for standard PyTorch models.

Path B - JAX Rewrite (Heavy CUDA): Rewrite in JAX/Flax using MaxText as reference. Use Pallas for custom kernels. Best performance.

Path C - vLLM-TPU (Inference only): vLLM now supports TPU backend. No code changes for supported models.

Won't work on TPUs: cuDNN-specific ops, custom CUDA C++ extensions, NCCL (use ICI instead), TensorRT quantization (use AQT).

Hard TPU Architecture
Explain TPU pod topology and how it affects model parallelism. What is the difference between single-host and multi-host TPU slices?

Topology

Single-Host: All chips attached to one VM (1x1, 2x2, 2x4). Simplest to program. Best for inference and small training.

Multi-Host: Chips span multiple VMs (4x4 and above). ICI connects across VM boundaries. Each VM runs SPMD. Best for large-scale training.

Medium TPU Provisioning
How do you provision TPU node pools in GKE? Explain Flex-start, Spot VMs, and reserved capacity options.

Options

On-Demand: Highest cost, highest availability. Use for production serving.

Spot VMs: 60-91% discount, can be preempted. Requires checkpointing.

Reserved (CUD): 1-3 year commitment, up to 57% off. Guaranteed capacity.

Flex-start (DWS): Queue request, provisioned when available. Better than on-demand pricing.

Calendar Mode (DWS): Book capacity for specific time window (up to 90 days).

Medium Cost Optimization
A customer is running inference on NVIDIA GPUs and wants to reduce costs. Would you recommend migrating to TPUs?

Recommend migration IF:

  • High-throughput, steady-state inference (not bursty)
  • Transformer-based models (LLMs, ViTs, diffusion)
  • Large batch sizes (TPUs lose efficiency at small batches)
  • Scale justifies effort (100+ GPUs worth)

Real-world results: Midjourney: $2.1M -> $700K/month (65% reduction). Character.AI: 3.8x cost improvement.

Don't migrate if: Heavy custom CUDA kernels, extremely latency-sensitive (less than 10ms), small scale (less than 10 GPUs), or zero JAX experience.

Medium AI Hypercomputer
What is Google AI Hypercomputer? How does it integrate TPUs, GPUs, storage, and networking?

AI Hypercomputer is NOT a product - it's a validated, integrated architecture that Google uses internally for Gemini training.

Four Layers

1. Hardware: TPUs (v5e, Trillium, Ironwood) + GPUs (A100, H100, H200). Jupiter network (1.6Tbps/node). GCS + Managed Lustre + Hyperdisk ML.

2. Software: JAX + XLA compiler. PyTorch/XLA. MaxText, vLLM-TPU, Pathways runtime.

3. Orchestration: GKE + Kueue + JobSet + XPK.

4. Consumption: DWS (Flex-start, Calendar), CUDs, Spot VMs.

Medium Scheduling
Explain how Dynamic Workload Scheduler works for GPU/TPU workloads in GKE.

Flex-start: Submit capacity request -> DWS queues -> provisions when available. Gang provisioning ensures all accelerators come up together.

Calendar Mode: Book capacity for specific time window (hours to 90 days). Guaranteed at scheduled time without CUD commitment.

Medium Troubleshooting
How do you troubleshoot a TPU training job stuck in Pending state?

Debug steps: 1. kubectl describe pod (check events). 2. Verify TPU quota. 3. Check node pool/autoscaler. 4. Validate topology match in nodeSelector. 5. Check Kueue ClusterQueue quotas. 6. Check DWS ProvisioningRequest status.

Hard Pathways
What is Pathways and how does it enable multi-slice TPU training with a single JAX client?

Pathways is a distributed runtime that lets a single JAX process orchestrate computation across multiple TPU slices (thousands of chips). Key components: IFRT Proxy Client/Server, Resource Manager, Pathways Workers. Supports automatic fault recovery from preemptions.

Hard Deployment Strategy
Implement a blue/green deployment strategy for a microservices application on GKE.

Deploy Green alongside Blue with different version labels. Service selector points to Blue. Cutover: update Service selector to Green. Rollback: switch back to Blue. Advanced: use GKE Gateway API with HTTPRoute traffic splitting for canary.

Medium Autoscaling
How do you configure autoscaling for GPU/TPU workloads in GKE?

HPA: Use custom metrics (GPU utilization, queue depth) not CPU. VPA: Right-size CPU/memory (not GPU requests). Cluster Autoscaler: Set min-nodes=0 for GPU pools. NAP: Auto-create GPU pools for pending pods.

Don't use HPA and VPA on same metric. Training job scaling via Kueue, not HPA.

Medium Security
What is Workload Identity in GKE and why is it preferred over service account keys?

Workload Identity creates trust between Kubernetes ServiceAccount (KSA) and GCP IAM Service Account (GSA). Pods impersonate GSA without credentials stored in cluster.

Why preferred: No key files (short-lived tokens). Least privilege per pod. Auditability. No Secret sprawl.

Medium Observability
How do you monitor and troubleshoot GPU training jobs in GKE?

Key metrics: GPU Utilization (DCGM) > 80%. GPU Memory at 90% alert. Training throughput (samples/sec). ECC errors (failing GPU).

Tools: DCGM Exporter DaemonSet + Cloud Monitoring + Cloud Logging to BigQuery.

Easy GKE Modes
GKE Standard vs GKE Autopilot - which for ML workloads?

Standard: ML training (precise GPU control, DaemonSets, custom machine types, node SSH).

Autopilot: ML inference, platform services (simpler ops, pay per pod, hardened security).

Hard Job Scheduling
Configure Kueue for multi-tenant GKE cluster with GPUs.

Components: ResourceFlavors (h100, l4) -> ClusterQueue (quotas, preemption) -> LocalQueues (per team) -> WorkloadPriorityClasses (prod > research > dev).

Key behaviors: Atomic admission (gang scheduling). BestEffortFIFO. Cohort borrowing. LowerPriority preemption.

Networking & Security

Key for Infrastructure Engineers • 6 Questions

Medium VPC Design
How do you design a network architecture on GCP for secure communication?

1. Shared VPC: Centralize in host project. Service projects share subnets.

2. Private clusters: No public IPs. Private Service Connect for APIs. Cloud NAT for outbound.

3. Defense in depth: Hierarchical firewall -> VPC firewall -> Network Policies -> VPC Service Controls.

4. Access: Identity-Aware Proxy for SSH (no bastion).

Medium Security Perimeter
What is VPC Service Controls and how does it enhance ML security?

VPC-SC creates security perimeter restricting data movement. Even with valid credentials, attackers can't exfiltrate to external buckets. Perimeter includes BigQuery, GCS, Vertex AI. Use ingress/egress rules for exceptions. Always test in dry-run first.

Medium Networking
Shared VPC vs VPC Peering - when to use each?

Shared VPC: Same org, centralized control. Recommended for multi-team ML platforms.

VPC Peering: Cross-org, third-party connections. NOT transitive (A-B, B-C ≠ A-C).

Hard GPU Networking
How does GPUDirect-TCPX improve A3 GPU instance performance?

GPUDirect-TCPX bypasses CPU in data path: GPU -> NIC -> Network -> NIC -> GPU. A3 Mega: 8x 200Gbps NICs = 1.6 Tbps total. gVNIC is prerequisite (high-perf virtual NIC, up to 200Gbps, Jumbo Frames). Critical for NCCL AllReduce in multi-node training.

Easy IAM
How do you grant least privilege access to ML training service account?

Grant only: storage.objectViewer (training data), storage.objectCreator (model artifacts), bigquery.dataViewer (features), monitoring.metricWriter (logs).

Never grant: roles/editor, roles/owner, roles/storage.admin.

Use IAM Conditions for resource-specific access. Workload Identity instead of JSON keys.

Hard Hybrid Security
How would you secure a hybrid cloud connecting on-prem GPU clusters to GCP?

Identity: Workforce Identity Federation (AD -> GCP IAM). Workload Identity Federation (on-prem K8s -> GCP APIs). No static credentials.

Network: Dedicated Interconnect + MACsec. Private Service Connect for API access.

Data: Cloud KMS + External Key Manager (keys in on-prem HSM). VPC Service Controls.

Easy Terraform State
How do you manage Terraform state in a team environment?

Backend: GCS with versioning, CMEK encryption, uniform bucket-level access.

Practices: State isolation per environment (prefix). Automatic locking. CI/CD-only applies. Read-only for developers.

Medium GitOps
How do you implement GitOps for GKE using Config Sync or ArgoCD?

Config Sync (GCP-native): Part of GKE Enterprise. Auto-sync from Git. Drift detection.

ArgoCD: Web UI, App-of-Apps pattern, multi-cluster support.

GitOps: Namespaces, RBAC, NetworkPolicies, Kueue queues. NOT GitOps: Training job submissions, secrets, node pool sizing.

Hard Troubleshooting
Customer's ML training keeps failing due to GPU preemption on Spot VMs. Design a fault-tolerant pipeline.

1. Periodic checkpointing: Save every N steps to GCS. Async writes.

2. SIGTERM handler: Emergency checkpoint in 25-second grace period.

3. Auto-resume via Kueue: restartPolicy: OnFailure, --resume-from-checkpoint.

4. Fallback: If preemption rate too high, switch to On-Demand.

Easy Communication

"GPUs are Swiss Army knives - versatile, work with everything. TPUs are power tools - purpose-built for AI, more efficient for that specific job."

Business framing: Cost (2-4x better cost-per-query). Ecosystem (PyTorch works, JAX ideal). Scale (TPUs + Jupiter network = system advantage). Availability (TPU = GCP only; GPU = anywhere).

Recommendation: "Start with a small TPU pilot for your largest inference workload. Benchmark cost-per-query. The data will make the decision obvious."
