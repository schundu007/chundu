Hard System Design
Design the infrastructure for an e-commerce shopping cart that handles 4 million concurrent users during prime time on GCP.

Approach Framework

At 4M concurrent users, we're looking at approximately 200K–400K requests/second at peak. The shopping cart is a write-heavy, session-critical service — any data loss means revenue loss. The design must prioritize low-latency reads/writes, horizontal scalability, and strong consistency for the cart state.

Key Design Decisions

1. Cart State in Memorystore Redis Cluster (not Spanner): Shopping carts are ephemeral, high-throughput, and latency-sensitive. A 30-shard Redis Cluster delivers sub-millisecond reads/writes with 100K+ ops/sec per shard. Cart data is serialized as Protobuf for compact storage. A background worker persists carts to Spanner every 60 seconds for durability.

2. Cloud Spanner for Orders and Inventory: Spanner provides externally consistent, globally distributed SQL with automatic sharding. For inventory (the hardest problem at this scale), Spanner's TrueTime-based transactions prevent overselling. We use a counter-based inventory model with row-level locks only during checkout, not during add-to-cart.

3. GKE Autopilot with Aggressive HPA: Autopilot eliminates node management overhead. HPA configured on custom metrics (requests-per-second from Cloud Monitoring) rather than CPU alone. Cart service scales to 1000 pods — each handling ~400 RPS. We use pod disruption budgets to ensure zero-downtime during scaling.

4. Pub/Sub as Event Bus: Every cart action (add, remove, update, abandon) emits an event to Pub/Sub. Downstream consumers handle analytics (BigQuery sink), recommendations (Vertex AI pipeline), and abandoned cart notifications (Cloud Functions to email/SMS). This decouples the hot path from analytics.

5. Cloud Armor + CDN: Cloud Armor provides L7 DDoS protection, geo-based access control, and rate limiting per IP (critical during flash sales). Cloud CDN caches product catalog pages and images with a 90%+ cache hit ratio, offloading ~3.6M of the 4M concurrent sessions from backend compute.

Capacity Math: 4M concurrent users x ~1 request every 10 sec = 400K RPS peak. Cart Service: 1000 pods x 400 RPS each = 400K RPS capacity. Redis: 30 shards x 15K writes/sec = 450K write ops/sec. Spanner: 10 nodes, ~40K writes/sec (checkout only, not cart updates).

Interview Tip: Always start with back-of-envelope math. Interviewers want to see you quantify the problem before drawing boxes. State your assumptions: avg session duration, click-to-request ratio, read vs. write ratio.

Phase 1: Assessment and Discovery

Workload Inventory: Catalog every training job — CUDA versions (11.x vs 12.x), framework dependencies (PyTorch/TensorFlow/JAX), multi-node communication patterns (NCCL AllReduce topology), and data pipeline sources. Map A100 40GB vs 80GB usage to determine GCP machine type: a2-highgpu-*g (A100 40GB) or a2-ultragpu-*g (A100 80GB).

Baseline Performance: Record training throughput (samples/sec), GPU utilization (nvidia-smi), and time-to-convergence for representative models. This becomes the acceptance criteria for the cloud migration.

Cost Model: Compare on-prem TCO (hardware depreciation, power, cooling, staff, facility) against GCP options. For sustained workloads: 1-year or 3-year Committed Use Discounts (CUDs) offer up to 57% off on-demand. For burst training: Spot VMs at 60-91% discount with checkpointing.

Phase 2: Landing Zone

Establish the GCP foundation using a Shared VPC architecture with a host project for networking and service projects for workloads. Deploy Dedicated Interconnect (10Gbps or 100Gbps) for hybrid data transfer during the transition period. Enable VPC Service Controls around BigQuery and GCS to prevent data exfiltration. Set up Terraform modules for reproducible GPU cluster provisioning.

Phase 3: Pilot Migration

Spin up a GKE cluster with a dedicated A2 GPU node pool. Install the NVIDIA GPU Operator for driver lifecycle management. Use Google's Deep Learning Container Images (pre-configured with CUDA, cuDNN, NCCL) to eliminate compatibility issues. Run the baselined training workload and tune NCCL environment variables (NCCL_NET_GDR_LEVEL, NCCL_P2P_LEVEL) for optimal multi-GPU communication.

Phase 4: Full Migration and Optimization

Migrate all training pipelines with automated checkpointing to GCS (every N steps). Implement Kueue for multi-tenant GPU scheduling and Dynamic Workload Scheduler (Flex-start mode) for cost-optimized GPU procurement. Consider upgrading to A3 Mega (H100) or A3 Ultra (H200) for workloads that benefit from newer GPU generations — the H100 provides ~3x training throughput over A100.

Data Transfer Strategy: For datasets less than 10TB, use gsutil parallel composite uploads. For 10-100TB, use Storage Transfer Service. For greater than 100TB, use Transfer Appliance. During the hybrid phase, keep data in both locations and validate checksums.

Interview Tip: Don't jump to "just containerize it." Show you understand the full picture — network connectivity, data gravity, CUDA compatibility, and the human/process side (team training, runbook migration, on-call handoff).

What Changes for Global

Load Balancing: Non-global uses a Regional External LB. Global uses a Global External HTTPS LB with Anycast IP — one IP routes users to the nearest healthy region automatically. Add Cloud CDN for static content caching at 180+ edge PoPs.

Compute: Non-global deploys a single GKE cluster spread across zones within one region (multi-zone for HA). Global deploys multiple GKE clusters across regions with Multi-Cluster Ingress (or Gateway API) for cross-cluster traffic management and Anthos Service Mesh for service-to-service observability.

Database: This is the biggest architectural change. Non-global can use Cloud SQL (regional HA with automatic failover). Global requires Cloud Spanner with a multi-region instance configuration (e.g., nam-eur-asia1) for globally consistent, low-latency reads via Spanner's TrueTime. Cloud SQL does not support multi-region writes.

Storage: Non-global uses Regional GCS buckets. Global uses Multi-Region or Dual-Region buckets with turbo replication (RPO of 15 minutes) for cross-region data availability.

DNS: Non-global uses standard A records. Global uses Cloud DNS with geolocation routing policies to direct users to the nearest regional endpoint.

Data Residency and Compliance: Global architectures must account for GDPR (EU data stays in EU), data sovereignty laws, and regional compliance (PCI-DSS scope may differ by region). Use Organization Policy constraints to enforce resource location restrictions.

Cost Implications: Global architectures are significantly more expensive. Spanner multi-region costs ~3x more than regional. Cross-region egress charges apply. Multi-cluster management overhead increases operational complexity.

Decision Framework: If users are within 100ms of a single region — stay single region, it's simpler and cheaper. If you need less than 200ms globally — go multi-region. If you need less than 50ms everywhere — deploy in 3+ regions with Spanner multi-region.

Design Decisions

1. Separate Node Pools by Workload Type: Training (GPU/TPU), inference (L4 GPUs), and platform services (CPU) each get dedicated node pools with appropriate machine types, taints, and tolerations. This prevents resource contention and enables independent scaling.

2. Kueue for Multi-Tenant Job Scheduling: Configure ClusterQueue with ResourceFlavors for GPU and TPU types. Create LocalQueues per team (research, production, experimentation). 3. 5. Spot VMs for Cost Optimization: GPU training node pool uses a mix: On-Demand nodes for baseline capacity, Spot VMs for burst. Configure PodDisruptionBudgets and checkpointing to handle Spot preemptions gracefully. This can reduce GPU costs by 60-91%.

6. Observability: Deploy DCGM Exporter DaemonSet for GPU metrics (utilization, memory, temperature, ECC errors). Cloud Monitoring custom dashboards for training throughput (samples/sec), GPU utilization, and job queue depth. Cloud Logging with structured log sinks to BigQuery for post-hoc analysis.

Interview Tip: Emphasize the separation of concerns — scheduling (Kueue), provisioning (autoscaler), placement (scheduler), and monitoring (DCGM/Cloud Monitoring). This shows you understand the layered architecture of a production ML platform.

Option A: GPU Path (Recommended for PyTorch Users)

Use 4 nodes of a3-megagpu-8g (32x H100 80GB GPUs = 2,560GB total HBM3). Apply Tensor Parallelism (TP=8) within each node (exploiting 900 GB/s NVLink) and Pipeline Parallelism (PP=4) across nodes (using GPUDirect-TCPX for 200Gbps inter-node). With FSDP (ZeRO-3 style), optimizer states are sharded across all 32 GPUs. Estimated training throughput: ~3,500-4,000 tokens/sec with BF16 mixed precision.

Option B: TPU Path (Recommended for JAX Users or Cost Optimization)

Use TPU v6e (Trillium) Multislice with 4 slices of 64 chips each (256 total chips). JAX's pjit with FSDP automatically shards the model and optimizer across all chips. ICI provides high-bandwidth communication within each slice; Multislice uses Data Center Network (DCN) across slices. Use Pathways for efficient multi-slice orchestration from a single JAX client. Cost advantage: TPU v6e offers approximately 1.8x better performance-per-dollar vs H100 for transformer training.

Key Configuration Details

Storage for Data Loading: Use Cloud Managed Lustre (1000 MB/s per TiB tier) as a high-speed cache layer. Training data resides in GCS; Lustre acts as a hot cache to prevent I/O bottlenecks. For datasets exceeding 10TB, this is critical — GCS FUSE alone cannot saturate 32+ GPUs.

Checkpointing: Use Orbax (JAX) or PyTorch distributed checkpointing to write async checkpoints every 100-200 steps directly to GCS. For a 70B model, each checkpoint is ~140GB; GCS handles this efficiently with parallel uploads. This protects against both node failures and Spot preemptions.

Cost Comparison (monthly, 24/7 training): 4x A3 Mega on-demand: ~$254K/month. 4x A3 Mega with 1yr CUD: ~$160K/month. 256-chip TPU v6e on-demand: ~$140K/month. TPU v6e with CUD: ~$80K/month. Spot GPUs (when available): ~$70K/month but with preemption risk.

RTO/RPO Targets by Component

Training Data (GCS): RPO = ~0 (use Dual-Region bucket with turbo replication, 15-minute RPO SLA). RTO = minutes (data is already in DR region). This is your most valuable asset — losing training data is unrecoverable.

Training Progress (Checkpoints): RPO = time since last checkpoint (typically 100-200 training steps = 10-30 minutes). RTO = time to provision GPUs + restore checkpoint (30-60 minutes). Store checkpoints in a dual-region GCS bucket. The latest checkpoint defines your recovery point — you replay from the last saved state.

Pipeline Metadata (Cloud SQL): RPO = seconds (async replication to read replica in DR region). RTO = 5-10 minutes (promote read replica to primary).

Model Artifacts (Artifact Registry): RPO = ~0 (multi-region registry). RTO = 0 (already available).

Overall DR Targets

RTO: 1-2 hours — dominated by GPU provisioning time in the DR region. Mitigation: pre-reserve a small GPU allocation via CUDs or use DWS Calendar mode to guarantee availability. RPO: 10-30 minutes — defined by checkpoint frequency. Mitigation: increase checkpoint frequency (trade training throughput for better RPO).

Failover Procedure

1. Cloud Monitoring detects primary region failure (GPU health checks, API latency SLOs). 2. Alert triggers PagerDuty and automated runbook. 3. Terraform apply in DR region provisions GPU instances from IaC. 4. Training pipeline restarts from latest GCS checkpoint. 5. Pipeline metadata promoted from Cloud SQL read replica. 6. DNS/routing updated if serving endpoints are affected.

Interview Tip: The key insight for ML DR is that training is inherently resumable via checkpointing. Unlike traditional DR where you need hot standbys, ML pipelines can cold-start from checkpoints. This dramatically reduces DR cost. Quantify the tradeoff: "We accept 30 minutes of lost training in exchange for not paying for idle GPUs in the DR region."

Key Design Decisions

1. Model Export Format: Train in JAX on TPUs, export to ONNX or SavedModel format for framework-agnostic inference. If inference uses NVIDIA GPUs, convert to TensorRT for optimal GPU utilization. Test model equivalence (output parity within tolerance) as part of the CI/CD pipeline.

2. Cross-Cloud Connectivity: Use HA VPN (up to 3Gbps per tunnel, encrypted IPSec) for control plane communication. For large model transfers (100GB+ checkpoints), use Storage Transfer Service (GCS to S3) — it's faster and cheaper than VPN for bulk data. Encrypt models at rest in both clouds using customer-managed encryption keys (CMEK).

3. Identity and Access (Zero Trust): Use Workload Identity Federation with OIDC to allow AWS/Azure workloads to authenticate to GCP services without service account keys. The inference cluster's IAM role maps to a GCP service account with read-only access to the model bucket. No long-lived credentials cross cloud boundaries.

4. Model Sync Pipeline: Automated pipeline: training completes → model validated → registered in Vertex AI Model Registry → exported to ONNX → copied to S3 via Storage Transfer Service → inference cluster pulls new version → canary deployment (10% traffic) → full rollout. This ensures the inference cloud always has the latest validated model.

5. Why Split Training/Inference? Common reasons: customer data residency (inference must run where users are), existing AWS/Azure contracts, TPU cost advantage for training but GPU ecosystem needed for inference (CUDA-optimized serving frameworks), or avoid vendor lock-in. This architecture lets you optimize each phase independently.

Security Checklist: Encrypt data in transit (TLS 1.3 between clouds). Encrypt data at rest (CMEK in both clouds). Workload Identity Federation (no static keys). VPC Service Controls on GCP (prevent data exfiltration from training environment). Network policies in both clusters. Audit logging in both clouds → centralized SIEM.

Network Integration

Dedicated Interconnect with 2x 10Gbps links (redundant) in two distinct edge availability domains for 99.99% SLA. Cloud Router with BGP provides dynamic route exchange — no static route maintenance. Ensure IP address ranges do not overlap between on-prem (e.g., 172.16.0.0/12) and GCP (10.0.0.0/8). Use Cloud DNS peering zones for bidirectional DNS resolution (on-prem can resolve *.googleapis.com; GCP can resolve internal.company.com).

Data Synchronization Strategy

Database CDC: Use Datastream for Change Data Capture from Oracle, MySQL, or PostgreSQL databases into BigQuery or Cloud SQL. This provides near-real-time synchronization (seconds of lag) without impacting source database performance.

File Sync: Use Storage Transfer Service with scheduled jobs to sync NFS shares to GCS buckets. For real-time file sync, deploy a Transfer Agent on-premises.

Event-Driven Sync: Deploy a Pub/Sub publisher on-premises (using the Pub/Sub client library) to emit business events. Cloud-side subscribers process events for real-time data integration, triggering Cloud Functions or Dataflow pipelines.

Identity Integration

Use Workforce Identity Federation to federate Active Directory identities into GCP via SAML 2.0. Users authenticate with their existing AD credentials to access Google Cloud Console and APIs. For workloads, use Workload Identity Federation to allow on-prem Kubernetes pods to authenticate to GCP services. Never export service account keys.

Security Posture

VPC Service Controls create a security perimeter around sensitive GCP services (BigQuery, GCS, Spanner) — even if credentials are compromised, data cannot be copied outside the perimeter. Cloud KMS with External Key Manager (EKM) allows encryption keys to remain in your on-premises HSM while encrypting GCP data — you retain key custody.

Interview Tip: The interviewer is looking for three things: (1) network connectivity design (Interconnect, not VPN for production), (2) identity federation (not service account keys), and (3) a specific data sync mechanism per data type (CDC for databases, Transfer Service for files, Pub/Sub for events). Don't give a one-size-fits-all answer.

Path A: PyTorch/XLA (Most Common Path)

For customers with standard PyTorch models (using nn.Module, standard ops, no custom CUDA C++ extensions), PyTorch/XLA is the lowest-friction path. Replace device placement from .cuda() to xm.xla_device(). Most PyTorch operations have XLA equivalents. The key consideration: PyTorch/XLA compiles computation graphs via XLA, so the model must be "trace-friendly" — dynamic control flow and tensor-dependent branching cause recompilation overhead. Use torch_xla.debug.metrics to identify compilation bottlenecks.

Path B: JAX Rewrite (Maximum Performance)

If the customer has heavy CUDA kernel dependencies (custom Triton kernels, CUTLASS, or handwritten CUDA C++), a JAX rewrite gives the best TPU performance. JAX's functional programming model aligns naturally with TPU's SPMD execution. Use Pallas (JAX's low-level kernel API) as a replacement for custom CUDA kernels. Reference implementations like MaxText (for LLMs) and MaxDiffusion (for diffusion models) provide production-quality starting points. This is the path companies like Anthropic, Lightricks, and Kakao have taken for peak performance.

Path C: vLLM-TPU (Inference Shortcut)

For inference-only migration, vLLM now supports TPU backend. The model's PyTorch code is automatically lowered through TorchAX into JAX, then compiled by XLA for TPU execution. No code changes needed for supported models. This is the fastest path to TPU inference, though model support is still growing.

CUDA Dependencies That Do NOT Work on TPUs: cuDNN-specific operations, custom CUDA C++ extensions (.cu files), NCCL (replaced by ICI for TPU-to-TPU communication), TensorRT quantization (use AQT instead), and any direct GPU memory management (cudaMalloc, etc.). These must be replaced or rewritten.

Topology Explained

TPU Pod Topology defines the physical arrangement of TPU chips connected via Inter-Chip Interconnect (ICI). For TPU v5e and v6e, topology is expressed as a 2D mesh (AxB), e.g., 2x4 = 8 chips, 4x8 = 32 chips, up to 16x16 = 256 chips (full pod). For v4 and Ironwood (v7), topology is 3D torus (AxBxC), e.g., 2x2x2 = 8 chips or 4x4x4 = 64 chips, enabling richer interconnect patterns for larger-scale communication.

Topology affects parallelism because ICI bandwidth within a topology is fixed — you must align your parallelism strategy to the physical chip layout. For a 4x8 topology: Tensor Parallelism (TP) should map to one dimension (e.g., the 4-chip dimension) where ICI is fastest, while FSDP/Data Parallelism maps to the other dimension (the 8-chip dimension). Misaligned sharding causes cross-topology communication that saturates ICI bandwidth.

Single-Host vs Multi-Host

Single-Host TPU Slice: All chips are attached to a single VM. Topologies: 1x1 (1 chip), 2x2 (4 chips), 2x4 (8 chips for v6e inference). Simplest to program — all chips visible as local devices in JAX/PyTorch. Best for: inference workloads, small model training, and development/testing.

Multi-Host TPU Slice: Chips span multiple VMs, each VM manages a subset of chips. Topologies: 4x4 and above. ICI connects chips across VM boundaries transparently. Each VM runs a copy of the training script in a multi-controller mode (SPMD). JAX handles cross-VM communication automatically via XLA. Best for: large-scale training, models that need more than 8 chips.

Key Difference for GKE: In GKE, multi-host slices are atomic — all nodes in the slice are treated as a single scheduling unit. You can't add or remove individual nodes. If one node fails, GKE recreates the entire slice. This affects failure handling: your training code must handle full-slice restarts via checkpoint resumption.

Option Details

On-Demand: Highest cost, highest availability. Use for production serving workloads that cannot tolerate interruption. No discount, no commitment.

Spot VMs: Massive discount (60-91%) but GCP can reclaim the resources at any time with 30-second warning. Essential requirement: frequent checkpointing. Medium Cost Optimization
A customer is running inference on NVIDIA GPUs and wants to reduce costs. Would you recommend migrating to TPUs? What factors would you consider?

Decision Framework

Yes, recommend migration IF:

  • The workload is high-throughput, steady-state inference (not bursty, one-off requests)
  • The model architecture is transformer-based (LLMs, vision transformers, diffusion models) — TPU MXUs are optimized for these
  • The batch size is large enough to keep TPU utilization high (TPUs lose efficiency at very small batch sizes)
  • The team can invest 4-8 weeks in migration and testing
  • The scale justifies the effort — typically 100+ GPUs worth of inference

Factors to Evaluate:

1. Model Architecture Fit: Transformer-based models are ideal. CNNs work well. Models with heavy branching, dynamic shapes, or sparse operations may not map efficiently to TPU MXUs. Custom CUDA kernels (FlashAttention, PagedAttention) have JAX equivalents (Pallas-based), but verify performance parity.

2. Quantization Path: On GPUs, customers typically use TensorRT or vLLM with INT8/FP8 quantization. On TPUs, use AQT (Accurate Quantized Training) for INT8 inference, or BF16 directly (TPU BF16 throughput is high enough that quantization is less critical than on GPUs).

3. Serving Framework: vLLM now supports TPU backend — this is the fastest migration path for LLM inference. JetStream (Google's open-source inference framework) provides optimized continuous batching for TPU. Triton Inference Server does not natively support TPU.

4. Latency Requirements: TPUs excel at throughput-oriented inference but may have slightly higher per-request latency than optimized GPU inference (TensorRT). For latency-critical applications (less than 50ms P99), benchmark carefully. For throughput-critical (chatbots, batch processing), TPUs typically win.

5. Cost Comparison (Real-World Data): Midjourney migrated from GPU to TPU v6e and reduced monthly inference spend from $2.1M to under $700K — a 65% reduction. Character.AI achieved 3.8x cost improvement. Waymark saw 4x lower cost than H100. These numbers are consistent across transformer inference at scale.

Do NOT recommend migration if: The customer's inference models use heavy custom CUDA kernels with no JAX equivalent, the workload is extremely latency-sensitive (less than 10ms), the scale is small (fewer than 10 GPUs — migration effort doesn't justify savings), or the team has zero JAX/XLA experience and no bandwidth to learn.

Medium Scheduling
Explain how Dynamic Workload Scheduler works for GPU/TPU workloads in GKE. When would you use it?

How DWS Works

Dynamic Workload Scheduler (DWS) is a resource management and job scheduling platform that sits between your workload and the accelerator capacity pool. Instead of you polling for GPU/TPU availability and manually provisioning VMs, DWS handles the allocation lifecycle.

Flex-start Mode: You submit a capacity request specifying: accelerator type (e.g., TPU v6e), topology (e.g., 4x8), and maximum wait time. DWS queues the request and automatically provisions VMs when capacity becomes available. Resources are provisioned simultaneously (gang provisioning) — all accelerators in your request come up together, which is critical for distributed training. You can also enable dynamic provisioning where resources trickle in over time (useful for inference scale-up). Integrated with GKE node pools via --enable-queued-provisioning and with Kueue via the ProvisioningRequest API.

Calendar Mode: You book accelerator capacity for a specific time window — from hours to 90 days. Similar to a flight booking: you select your start date, duration, and accelerator type. Calendar mode provides guaranteed capacity at the scheduled time without the long-term commitment of CUDs. Now GA as of December 2025. Supports TPUs and A3/A4 GPU VMs.

When to Use DWS

  • Flex-start: Training jobs with flexible start times, batch fine-tuning, experimentation runs where you need GPUs/TPUs but can wait hours for them
  • Calendar mode: Planned training runs with known schedules, demo/benchmark timelines, short-term projects (1-90 days) where CUDs are overkill
  • Do NOT use DWS: Real-time inference (need instant provisioning), very short jobs (less than 1 hour — provisioning overhead dominates), or when you already have reserved capacity via CUDs

What Pathways Does

Pathways is a distributed runtime system that allows a single JAX Python process to orchestrate computation across multiple TPU slices — potentially spanning thousands of chips. Without Pathways, multi-slice training requires a separate JAX process per slice (multi-controller mode) with explicit cross-slice communication. With Pathways, the user writes code as if they have one giant supercomputer, and Pathways handles distribution, placement, fault tolerance, and cross-slice communication.

Key Components

IFRT Proxy Client: A JAX backend implementation that intercepts JAX operations and routes them through the Pathways system instead of executing locally. This is transparent to user code — you write standard JAX, and the IFRT proxy handles the distribution.

IFRT Proxy Server: Receives High-Level Operations (HLOs) from the client, coordinates with the resource manager for chip allocation, and dispatches compiled programs to workers.

Resource Manager: Manages TPU chip allocation across slices. Determines where compiled programs should execute based on chip availability and topology constraints.

Pathways Workers: Run on TPU VMs. They receive compiled XLA executables and perform the actual computation. Workers communicate with each other via ICI (within a slice) and DCN (across slices).

How Multi-Slice Training Works

In a multi-slice configuration (e.g., 3x 256-chip slices = 768 chips total), Pathways extends XLA's SPMD parallelism across slice boundaries. Within each slice, ICI provides high-bandwidth communication for tensor/FSDP parallelism. Across slices, DCN handles data parallelism gradient synchronization. Pathways optimizes cross-slice collectives to minimize DCN transfers — for example, using reduce-scatter within slice + all-reduce across slices patterns that Google developed from scaling Gemini training.

Fault Tolerance: Pathways supports automatic recovery from preemptions and chip failures. If a slice goes down, Pathways can checkpoint, re-allocate resources, and resume training without the user needing to handle failure logic. This is critical for multi-day training runs on Spot TPUs.

EasyFundamentals
Explain the difference between virtualization and containerization. (Real Glassdoor question)

Answer

Virtualization creates complete virtual machines, each with its own OS kernel, running on a hypervisor (e.g., KVM, VMware). Each VM includes a full guest OS (typically 1-20 GB), boots in minutes, and provides strong isolation via hardware-level separation. Think of it as separate houses — each with its own foundation, plumbing, and electrical.

Containerization packages an application and its dependencies into a lightweight, isolated process that shares the host OS kernel. Containers (via Docker/containerd) are typically 10-500 MB, start in seconds, and provide process-level isolation via Linux namespaces and cgroups. Think of it as apartments in the same building — sharing the foundation but with separate living spaces.

HardGPU Networking
How does GPUDirect-TCPX improve network performance for A3 GPU instances? What about gVNIC?

Answer

GPUDirect-TCPX is a technology that enables GPU-to-GPU communication across network nodes by bypassing the CPU in the data path. In standard networking, data flows: GPU → Host CPU → NIC → Network → NIC → Host CPU → GPU. With GPUDirect-TCPX, the data path is: GPU → NIC → Network → NIC → GPU — the CPU is removed from the critical path, reducing latency and freeing CPU for data preprocessing.

On A3 Mega instances (8x H100), each node has 8x 200Gbps NICs (one per GPU), providing 1.6 Tbps total bisectional bandwidth. GPUDirect-TCPX ensures this bandwidth is directly accessible from the GPU without CPU bottleneck. This is critical for multi-node training where NCCL AllReduce operations dominate communication. Without GPUDirect-TCPX, inter-node gradient synchronization would be 2-3x slower.

gVNIC (Google Virtual NIC) is a high-performance virtual network interface that replaces the older VirtIO-Net. gVNIC provides: higher bandwidth (up to 200Gbps per NIC), better packet processing with less CPU overhead, support for Jumbo Frames (MTU 8896), and hardware-assisted flow steering. gVNIC is a prerequisite for GPUDirect-TCPX — you must use gVNIC-enabled machine images on A3 instances.

Configuration: On A3 instances, gVNIC and GPUDirect-TCPX are configured automatically when using Google's Deep Learning VM images or DLVM container images. For custom images, install the gve driver and gpu-rdma packages. Set NCCL environment variables: NCCL_NET=gIB, NCCL_GPUDIRECTTCPX_FORCE_ACK=0.

For a 70B model on H100s: checkpoint every 100-200 steps (~140GB per checkpoint). Use async checkpointing (Orbax for JAX, torch.save with background thread for PyTorch) so training doesn't pause during writes. Store in GCS with gsutil -m cp parallel uploads.

2. SIGTERM Handler: Spot VMs receive a 30-second warning before preemption. Register a SIGTERM handler that triggers an emergency checkpoint — save whatever state you can in 25 seconds. Even a partial checkpoint is better than losing all progress since the last periodic checkpoint.
