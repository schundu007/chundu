ROCm / TheRock · CI internals

The shadow build system, explained — file by file

TheRock's real build orchestrator isn't YAML — it's ~46 Python scripts that compute GitHub Actions matrices at runtime and hand them to workflows as JSON. This is a field guide to the files that matter: why each exists, what's inside, how it wires to the others, and where it's strong or brittle — plus an honest read on which build/CI tools would actually cut the file sprawl.

Read from the public, MIT-licensed ROCm/TheRock repo, analyzed on main @ f44bc6af (July 2026). The repo is mid-migration — a multi_arch_* path is superseding older code — so specifics drift. Descriptions are grounded in the actual source; they are explanations, not excerpts.

How a build gets decided

Fix the current practice — no migration

The most useful alternative to today's ~46 scripts + 54 fromJSON YAMLs isn't a new vendor — it's hardening the practice you already have. Every option below is adoptable now, stays 100% on GitHub Actions, and needs no big migration. Start here, before the bigger bets in §04.

05

Other tools worth knowing

Beyond buildx / BuildKite / Bazel, these are the credible options in the same space — grouped by which of the two problems they actually attack. Fit ratings are for TheRock specifically (a k8s-hosted, CMake-from-source, multi-arch superbuild), not in general.

Concretely — if TheRock adopted Dagger. The decision logic doesn't disappear; it moves from ~46 loose scripts + 54 fromJSON YAMLs into one typed module you can run on your laptop. GitHub Actions shrinks to a thin trigger.

Today46 Python scripts serialize JSON → GITHUB_OUTPUT; 54 workflow YAMLs re-parse it via fromJSON(). A matrix bug shows up at 3am in the nightly, not at review. Answering “what will CI do with my PR?” means push-and-watch.
With DaggerOne module exposes ci(event) that returns the typed job graph. dagger call ci --event=pr.json prints the exact matrix locally in seconds. The workflow becomes a few lines that just invoke it.

Shrinks: most of the 54 orchestration YAMLs collapse to thin triggers; the fromJSON string-injection seam vanishes; the generator becomes directly unit-testable and locally runnable.  Doesn't: GPU-hardware runners are still required; the decision logic still has to exist (now as typed code); and it's a real migration — best done incrementally by first wrapping the existing scripts as Dagger functions, not a big-bang rewrite.
