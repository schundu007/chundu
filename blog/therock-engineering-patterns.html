Five CI patterns worth stealing from a GPU superbuild

ROCm/TheRock assembles a compiler toolchain, HIP/OpenCL/core runtime, math and ML libraries, and profilers into one CMake superbuild across many GPU targets and two operating systems. Read its public .github/ and build_tools/ trees and a handful of reusable engineering patterns fall out — the kind you can carry into any large multi-repo build. Here are five, with the trade-offs each one is really solving.

01

Centralize the CI, don't triplicate it

Problem — the same matrix logic copy-pasted into three repos drifts apart

A superproject and its component repos each need to run "the same" CI: which GPU families build, on which OS, against which Python versions, producing which artifacts, on which trigger. The naïve approach copies the matrix generator and workflow files into every repo — and they immediately begin to diverge. The pattern in the repo is a single source of truth: the GPU-family matrix and the CI-config logic live once in the superproject, and component repos call the superproject's reusable workflows on both Linux and Windows, parameterized by a small "which external repo am I building?" detection step.

The interesting part isn't the centralization — it's how it was rolled out. The change history shows a large single-shot consolidation that was reverted within a day, then re-landed as a sequence of small, self-contained, independently reviewable pieces: first just the repo-detection module (with its own unit tests), then a naming-standardization change, and only later the workflow rewiring. The lesson is written into the commit graph.

Why the revert mattersThe revert wasn't a failure — it followed the same norm LLVM documents: if a reviewer raises a concern that would have blocked a merge, you revert first and discuss, rather than defend a landed change. A 2,000-line "big bang" is unreviewable after the fact; five 200–700-line PRs are. Blast radius is a design constraint, not an afterthought.
Take-away: one matrix authority + reusable workflows kills cross-repo drift — but land it in increments a reviewer can actually hold in their head, and treat "revert-first" as cheap insurance, not defeat.
02

Migrate test scripts to the repo that owns the code

Problem — test runners live in the CI repo, far from the code they test

~35 pytest runner scripts sat in the superproject, but the binaries they exercise are built from source in the component repos. That's a structural mismatch: component teams submit test changes to a different repo, tests can't co-evolve atomically with the code, and CODEOWNERS can't align. The repo tackles this with a written RFC — three placement options weighed with explicit pros/cons, a chosen option that minimizes the number of CODEOWNERS approvals (one entry per repo, not one per project), and a longer-term option deferred to a follow-on RFC. Migration is a documented six-step plan: inventory → copy → finalize → switch CI refs → validate → delete.

The subtle engineering is path resolution. A script that used to live in the CI repo hardcoded a "walk up three parents" to find shared helpers — which breaks the moment it moves. The portable fix makes a script work both as a submodule of the superproject and standalone:

THEROCK_DIR = Path(
    os.environ.get("THEROCK_DIR") or SCRIPT_DIR.parent.parent.parent
).resolve()

...and one script needed its header reordered so THEROCK_DIR is defined before the sys.path insertion that depends on it — the kind of ordering bug you only find by actually running the move. A premature "switch CI to the new locations" step was reverted on a pre-declared blocker: you can't point CI at the new paths until the component submodule refs carrying those files are bumped into the superproject first.

Take-away: ownership follows the code. Move tests to the repo that builds the binary, gate the move on CODEOWNERS reality, and design one path-resolution shim that works in both checkout topologies before you move a single file.
03

Patch-back: keeping a monorepo and its downstreams in sync

Problem — code lives in a monorepo but must mirror to per-component downstream repos

When many libraries are consolidated into one monorepo but still need to flow back to individual downstream repositories, you need bidirectional sync tooling. The component repos carry a "PR patch-back" / merge-sync system built on GitHub Actions and git subtree/patch mechanics. What makes it real engineering is the edge cases it explicitly handles:

  • Squash vs. rebase vs. merge commits — the patch generator has to reconstruct the right change set depending on how the source PR was merged.
  • Empty patches and path-filtered commits — no-op mirrors must be detected and skipped, not force-pushed.
  • git am over git apply — switched to preserve authorship/metadata and to fail loudly on conflict instead of silently misapplying.
  • Per-file patches and subtree-change detection so adds/moves mirror correctly, with an auto-push escape hatch when subtree performance regresses.
Why it's hardThis is distributed-consistency work disguised as CI plumbing. The failure mode isn't a red build — it's two repos silently disagreeing about the same file. Every branch in this logic exists because some real merge shape produced a wrong or empty mirror once.
Take-away: if you mirror code between repos, the merge shape (squash/rebase/merge) is the hard part. Design for all three up front, make empty-diff a first-class case, and prefer tooling that fails loudly.
04

Reuse baselines via a build-topology reverse index

Problem — CI rebuilds things that a prior run already produced

At superbuild scale, the expensive question is "what can I not rebuild?" The repo adds helpers that build a reverse index over the build topology — mapping outputs back to the inputs and jobs that produce them — and a baseline run selection layer that picks a suitable prior run to reuse artifacts from, then wires that selection into multi-arch dispatch. Instead of every push triggering a full multi-architecture rebuild, a run can validate which build jobs are eligible to reuse a baseline and only do the delta.

This is the same instinct as content-addressed caching, lifted from the object level up to the job/topology level: the graph itself tells you the safe reuse boundary, so CI cost tracks the size of the change, not the size of the tree.

Take-away: model your build as a graph you can query. Once outputs point back to inputs, "reuse the last good baseline and rebuild only the delta" becomes a lookup instead of a guess.
05

Python is the build system; YAML is just the output format

Problem — build logic that outgrows what YAML can express

The decisive architectural choice is that the matrix — GPU families × OS × Python version × artifact × trigger — is not written in YAML. It's generated by a large body of Python in build_tools/github_actions/ and injected into GitHub Actions via fromJSON(). YAML is the output; Python is the build system; Actions is an executor of whatever JSON it's handed:

configure:
  outputs:
    matrix: ${{ steps.gen.outputs.matrix }}
  steps:
    - id: gen
      run: python3 build_tools/github_actions/fetch_build_configurations.py \
             --event-name "${{ github.event_name }}" \
             --github-output "$GITHUB_OUTPUT"

build:
  needs: configure
  strategy:
    matrix: ${{ fromJson(needs.configure.outputs.matrix) }}

The payoff is testability: a matrix generator in Python gets unit tests; a matrix hand-written in YAML gets copy-paste. Across the CI tooling the pattern holds — new modules ship with their tests, which is exactly why the increment-and-revert discipline in pattern 01 is even possible.

Take-away: when your matrix needs conditionals, treat CI config as a program: generate JSON from tested code and hand it to the runner. YAML is a serialization target, not a place to put logic.
+

The through-line

Underneath the five patterns is one temperament: protect the mainline, and make change reviewable. Timeouts are tuned rather than maxed out; runner-health checks gate test jobs; Windows and WSL builds are exercised in dual modes (a strict pass/fail lane and an informational one) so a flaky new platform can't block the trunk; and reverts — including of one's own large changes — are treated as a normal, cheap tool rather than an admission of defeat. None of that is glamorous. All of it is what keeps a superbuild green.
