CI / Build Engineering · Patterns, not people

Five CI ideas worth borrowing from a giant GPU build

Every time someone changes code on a big project, an automated system rebuilds it and re-runs the tests. That system is called CI (continuous integration). ROCm/TheRock is AMD's open-source project that builds the entire GPU software stack — the compiler, the runtime, the math and AI libraries — in one giant build, for many GPU models, on both Linux and Windows. Read the public code behind its CI and five smart, reusable ideas jump out. They aren't specific to AMD; you could use any of them on any large project that is split across many separate code repositories (repos). Here are the five, and the real problem each one solves.

01

Keep your build rules in one place — don't copy them everywhere

Problem it solves — most projects copy the same build rules into every repo, and the copies slowly drift apart

A big main project (the superproject) and its smaller sub-projects (the component repos) all need to run the same checks whenever code changes: which GPU models to build for, on which operating system, with which Python versions, what to produce, and what should trigger it. That whole set of combinations is called the build matrix. The easy but wrong way is to copy the matrix and the build scripts into every repo — and then watch the copies go out of sync.

How TheRock does it: it keeps one copy of the rules in the main project, and every sub-project simply calls those shared rules — on both Linux and Windows. Each sub-project only adds one tiny step that answers, "which project am I building right now?"

The clever part isn't the idea itself — it's how they shipped it. The git history tells a little story. Someone first made this change as one huge commit. It was undone (reverted) the very next day. Then it came back in small, bite-sized pieces that a reviewer could actually check one at a time: first just the tiny "which project am I?" detector, with its own tests; then a simple renaming clean-up; and only at the end the real rewiring of the build. The lesson is sitting right there in the commit history.

Why undoing it was the right callUndoing the change wasn't an embarrassment — it's a healthy habit that big projects like LLVM recommend. If a reviewer spots a real problem after a change has already gone in, you undo it first and then discuss, instead of arguing to keep it. A single 2,000-line change is almost impossible to review once it has landed; five changes of 200–700 lines each are easy. Put simply: how much can break at once is something you plan for on purpose, not something you discover later.
The lesson: keep one place that owns the build rules and let everyone share it — that stops the copies from drifting. But ship the change in small steps a reviewer can hold in their head, and treat "undo first, discuss later" as cheap insurance, not as failure.
02

Move test scripts into the repo that owns the code

Problem it solves — in most multi-repo projects the test scripts end up in one repo while the code they test lives in another

That split is awkward: a team that owns some code has to send its test changes to a different repo, the tests and the code can't change together in one go, and the "who must approve changes to this file" list (GitHub calls this CODEOWNERS) can't line up with the real owners.

How TheRock does it: about 35 test scripts lived in the main project while the programs they test are built inside the separate component repos — so the move started with a written proposal, an RFC, a short design document the team reviews before anyone writes code. The proposal lists three ways to place the tests, with honest pros and cons, and picks the one that needs the fewest approvals to merge (one owner entry per repo, not one per sub-part). A bigger future clean-up is written down but left for later. The move itself follows a clear six-step checklist: list the files → copy them → tidy them up → point CI at the new spot → check everything still works → delete the old copies.

The subtle part is teaching each script how to find its helper files. An old script simply assumed "go up three folders" to reach the shared helpers — which breaks the instant you move it somewhere else. The fix makes a script work both ways: when it runs inside the big project, and when it runs on its own:

THEROCK_DIR = Path(
    os.environ.get("THEROCK_DIR") or SCRIPT_DIR.parent.parent.parent
).resolve()

...and one script had a subtle ordering bug: it tried to use a folder path before it had worked out what that path was, so two lines had to be swapped. You only catch that by really running the move, not just reading the code. There was also a false start — someone pointed CI at the new file locations too early. That was undone on purpose, because of a rule the team had written down in advance: you can't switch CI to the new paths until the sub-projects that carry those files have first been pulled into the main project.

The lesson: tests belong with the code they check. Move them into the repo that actually builds the program, make sure the "who approves this" list still makes sense, and design one small "find my files" helper that works whether the script runs on its own or inside the big project — before you move a single file.
03

Keep the big shared repo and its smaller copies in sync — automatically

Problem it solves — code lives in one big shared repo, but copies still have to flow out to smaller separate repos

When lots of libraries are pulled together into one big shared repository (a monorepo) but each library still needs its own separate public repo, you need tooling that keeps both in sync — with changes flowing in both directions. Doing it by hand means two repositories quietly drifting apart.

How TheRock does it: its repos carry a "patch-back" system, built on GitHub Actions and git's copy-changes-between-repos features. What makes it real engineering is the messy real-world cases it deliberately handles:

  • Different merge styles (squash, rebase, or a plain merge) — how a change was combined into the source repo changes how you have to rebuild it for the copy, so the tool handles all three.
  • Changes that copy to nothing — sometimes a change would leave the copy identical. The tool spots these "nothing to do" cases and skips them, instead of pushing an empty update.
  • Using git am instead of git apply — a small switch that keeps the original author's name and details, and fails loudly on a conflict instead of quietly applying the wrong thing.
  • File-by-file copying and change detection, so new and moved files land correctly — plus a manual fallback for when the automatic method gets too slow.
Why it's hardThis looks like plumbing, but it's really about keeping two copies of the truth in agreement. The scary failure isn't a red "build failed" light — it's two repositories quietly disagreeing about the same file, with nobody noticing. Every extra branch in this code exists because some real merge, at some point, produced a wrong or empty copy.
The lesson: if you copy code between repos, the way a change was merged (squash / rebase / merge) is the hard part. Plan for all three from the start, treat "nothing changed" as a normal case, and prefer tools that shout when something is wrong rather than fail quietly.
04

Reuse an earlier build instead of rebuilding everything

Problem it solves — CI keeps rebuilding things it already built in an earlier run

At this scale, the money question is: "what can I avoid rebuilding?" Most CI systems answer it badly — every push kicks off a full rebuild for every target, even when almost nothing changed.

How TheRock does it: it adds helpers that build a lookup table (a reverse index) over the build's dependency map — it connects each finished output back to the inputs and jobs that made it. On top of that sits a baseline picker that chooses a good earlier run to reuse results from, and feeds that choice into the build. A run then works out which jobs are safe to reuse from that earlier run, and rebuilds only what actually changed.

It's the same idea as ordinary caching — "don't redo work you have already done" — but moved up a level: instead of reusing individual files, it reuses whole build jobs. The dependency map itself tells you how far a change can reach, so the cost of a CI run matches the size of your change, not the size of the whole project.

The lesson: treat your build as a map you can ask questions of. Once every output points back to its inputs, "reuse the last good run and rebuild only what changed" becomes a simple lookup instead of a guess.
05

Generate the build matrix from tested code — let YAML just print it

Problem it solves — hand-written build rules outgrow what YAML can express, and nobody can test them

Once your build has real "if this, then that" logic, YAML (the simple config format GitHub Actions uses) stops being a good home for it: a matrix typed out by hand can only be copy-pasted and hoped over.

How TheRock does it: the build matrix — GPU model × operating system × Python version × output × trigger — is not typed out by hand at all. Instead, a large amount of Python in build_tools/github_actions/ generates that list and hands it to GitHub Actions as JSON, through a function called fromJSON(). Put plainly: Python is the brain that decides what to build; YAML is just the printout; and GitHub Actions simply runs whatever list it is given:

configure:
  outputs:
    matrix: ${{ steps.gen.outputs.matrix }}
  steps:
    - id: gen
      run: python3 build_tools/github_actions/fetch_build_configurations.py \
             --event-name "${{ github.event_name }}" \
             --github-output "$GITHUB_OUTPUT"

build:
  needs: configure
  strategy:
    matrix: ${{ fromJson(needs.configure.outputs.matrix) }}

Why do it this way? Testing. A build list written in Python can have proper automated tests. Across all of TheRock's CI tooling the same habit shows up — new pieces arrive together with their tests. That is exactly what makes the small-steps-and-undo discipline from pattern 1 possible in the first place.

The lesson: when your build needs "if this, then that" logic, treat your CI config as a real program — generate the JSON from tested code and hand it to the runner. YAML is a place to print results, not a place to put your thinking.
+

The through-line

Under all five patterns is one simple attitude: protect the main branch, and make every change easy to review. Time limits are set thoughtfully instead of cranked to the maximum. Test jobs only start once the machine they will run on has been checked and is healthy. Windows and WSL (Windows' built-in Linux) builds run in two lanes at once — a strict lane that can block a release, and an "FYI only" lane — so a shaky new platform can't hold everyone else up. And undoing changes, even your own big ones, is treated as a normal, cheap tool rather than a defeat. None of this is glamorous. All of it is what keeps a build this big passing, day after day.
