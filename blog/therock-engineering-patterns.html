How big software gets built · About the code, not the people

Five build lessons from a giant GPU project

On a big software project, a robot does the building. Every time somebody changes the code, a computer builds the whole thing again and runs all the tests. People call this CI, short for continuous integration.

ROCm/TheRock is AMD's open project. It builds all of AMD's GPU software in one go — the compiler, the parts that run your program, and the maths and AI libraries. It does this for many models of GPU, on both Windows and Linux.

Read the code behind its CI and five good ideas jump out. None of them are only for AMD. You could use any of them on any big project whose code is split across many separate folders. Each of those folders is called a repo.

Here are the five, and the real problem each one fixes.

01

Keep your build rules in one place — don't copy them everywhere

The problem it fixes — most teams copy the same build rules into every repo, and the copies slowly drift apart

A big project has one main repo and several smaller ones. All of them have to run the same checks when the code changes. Which GPU models to build for. Which operating system. Which version of Python. What to make at the end. And what should set the whole thing off.

Put all those choices together in one list and you get what people call the build matrix.

The easy thing is to copy that list into every repo. It is also the wrong thing. Copies get edited one at a time. After a few months no two are quite the same, and nobody is sure which one is right.

How TheRock does it: it keeps one copy of the rules, in the main repo. Every smaller repo just calls that one copy. This works the same way on Linux and on Windows. Each smaller repo adds only one little step of its own, which answers a single question: which project am I building right now?

Now the interesting bit — how they made the change. The history of the code tells a small story. Someone first did the whole thing in one huge change. The next day it was taken back out. (Taking a change back out is called a revert.) Then it came back in small pieces, one at a time, each small enough for another person to actually read. First the little "which project am I?" checker, with its own tests. Then a simple tidy-up of some names. The real rewiring came last.

Why taking it back out was the right callPulling the change out was not a telling-off. It is a normal habit, and big projects such as LLVM tell people to do it. If somebody spots a real problem after a change has gone in, you take it out first and talk about it after. That way nobody has to argue while the broken thing sits there.

One change of 2,000 lines is nearly impossible to check once it has landed. Five changes of 200 to 700 lines each are easy. So you decide up front how much is allowed to break at once. You don't find out the hard way later.
The lesson: one place owns the build rules and everybody else borrows them. That stops the copies drifting apart. Make the change in small steps that another person can hold in their head. And treat "take it out first, talk about it after" as cheap insurance, not as failure.
02

Move test scripts into the repo that owns the code

The problem it fixes — in most split-up projects the tests sit in one repo and the code they test sits in another

That split makes life hard. A team that owns some code has to send its test changes to somebody else's repo. The test and the code cannot change together in one go. And the list of people who must say yes to a change (GitHub calls this list CODEOWNERS) cannot match the people who really own the code.

How TheRock does it: about 35 test scripts sat in the main repo, while the programs they test are built in the smaller repos. The move started with a written plan. This kind of plan is called an RFC — a short document the team reads and argues over before anyone writes any code.

The plan lists three ways to place the tests. It gives the good and bad points of each one, honestly. Then it picks the one that needs the fewest people to say yes: one owner listed per repo, rather than one per small part. A bigger tidy-up is written down and left for another day.

The move itself follows a six-step list: write down the files → copy them → tidy them → point CI at the new place → check nothing broke → delete the old copies.

The tricky part is teaching each script how to find its helper files. An old script simply assumed "go up three folders" to find the shared helpers. That breaks the moment you move it. The fix makes a script work both ways — when it runs inside the big project, and when it runs on its own:

THEROCK_DIR = Path(
    os.environ.get("THEROCK_DIR") or SCRIPT_DIR.parent.parent.parent
).resolve()

One script also had a small order bug. It used a folder path before it had worked out what that path was. Two lines had to swap places. You only find that by really doing the move, not by reading the code.

There was one false start too. Someone pointed CI at the new file places too early. That was taken back out on purpose, because of a rule the team had written down beforehand: you cannot point CI at the new places until the smaller repos holding those files have been pulled into the main one first.

The lesson: tests belong next to the code they check. Move them into the repo that actually builds the program. Make sure the "who says yes to this" list still makes sense. And work out your small "how do I find my files" helper before you move a single file.
03

Keep the big repo and its copies in step — automatically

The problem it fixes — the code sits in one big shared repo, but copies still have to go out to smaller separate ones

Sometimes many libraries are kept together in one big repo. People call that a monorepo. But each library still needs a public repo of its own as well. So changes have to travel both ways, and something has to keep the two sides saying the same thing. Do it by hand and they quietly drift apart.

How TheRock does it: its repos carry a "patch-back" system, built on GitHub Actions and on git's own tools for copying changes between repos. What makes it real work, rather than a toy, is the messy cases it deals with on purpose:

  • Three ways of joining changes together (squash, rebase, or a plain merge) — how a change was folded into the first repo changes how you have to rebuild it for the copy. The tool handles all three.
  • Changes that copy to nothing — sometimes a change would leave the copy exactly as it already was. The tool spots that and does nothing, instead of pushing an empty update.
  • Using git am instead of git apply — a small swap. It keeps the name of whoever wrote the change, and it stops loudly when two changes clash, instead of quietly doing the wrong thing.
  • Copying file by file, so new and moved files land in the right place — plus a way to do it by hand for when the automatic way gets too slow.
Why this is hardThis looks like boring plumbing. It is really about keeping two copies of the truth saying the same thing. The frightening failure is not a red "build failed" light. It is two repos quietly disagreeing about the same file while nobody notices. Every extra branch in this code is there because some real merge, one day, made a wrong or empty copy.
The lesson: if you copy code between repos, the way a change was joined in (squash, rebase or merge) is the hard part. Plan for all three from the start. Treat "nothing changed" as a normal thing that happens. And pick tools that shout when something is wrong rather than fail in silence.
04

Reuse the last build instead of doing it all again

The problem it fixes — CI keeps building things it already built last time

At this size, the question that costs real money is simple: what can I avoid building again? Most CI answers it badly. Every change starts a full rebuild of everything, even when almost nothing is different.

How TheRock does it: a build knows which parts need which other parts. That is its map. TheRock adds helpers that turn the map around, so you can also ask it the opposite question: for this finished piece, what went into it, and which job made it? A turned-around map like this is called a reverse index.

On top of it sits a picker. It chooses a good earlier run to borrow results from, and hands that choice to the build. The run then works out which jobs are safe to borrow, and builds again only what really changed.

This is the same idea as ordinary caching — don't redo work you have already done — but one level up. Instead of reusing single files, it reuses whole jobs. And the map tells you how far a change can reach. So the cost of a run matches the size of your change, not the size of the project.

The lesson: treat your build as a map you can ask questions of. Once every finished piece points back at what made it, "borrow the last good run and rebuild only what changed" becomes a simple look-up instead of a guess.
05

Let a real program decide what to build — the config file just prints the answer

The problem it fixes — hand-written build rules grow too clever for a config file, and nobody can test them

GitHub Actions is set up using a simple config format called YAML. YAML is fine for plain lists. It is bad at "if this, then that". And a list typed out by hand cannot be tested. It can only be copied and hoped over.

How TheRock does it: the build matrix — GPU model × operating system × Python version × what to make × what sets it off — is not typed out by hand at all. A large amount of Python in build_tools/github_actions/ works the list out and hands it over as JSON, through a function called fromJSON().

Plainly: Python is the brain that decides what to build. YAML is just the paper it gets printed on. GitHub Actions runs whatever list it is handed:

configure:
  outputs:
    matrix: ${{ steps.gen.outputs.matrix }}
  steps:
    - id: gen
      run: python3 build_tools/github_actions/fetch_build_configurations.py \
             --event-name "${{ github.event_name }}" \
             --github-output "$GITHUB_OUTPUT"

build:
  needs: configure
  strategy:
    matrix: ${{ fromJson(needs.configure.outputs.matrix) }}

Why do it this way? Because you can test it. A build list written as a program can have real tests run against it. And the same habit shows up all through TheRock's CI tools: new pieces arrive together with their tests. That is what makes the small-steps-and-take-it-back-out habit from lesson 1 possible at all.

The lesson: when your build needs "if this, then that" thinking, treat your CI setup as a real program. Work the list out in code you can test, then hand it over. A config file is a place to print an answer, not a place to keep your thinking.
+

What ties all five together

Under all five is one simple attitude: look after the main copy of the code, and make every change easy to read.

Time limits are set with thought, not just turned up to the highest number. Test jobs only start once the machine they will run on has been checked and found healthy. Windows builds run in two lanes at the same time — a strict lane that can hold up a release, and a "just so you know" lane — so a shaky new platform cannot block everybody else. And taking a change back out, even your own big one, is treated as a normal cheap tool rather than a defeat.

None of this is exciting. All of it is what keeps a build this big passing, day after day.
